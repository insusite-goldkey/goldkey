## 제6조 [Bright Corporate 스타일 — 전역 배경 및 텍스트]

앱 전체 배경(`.stApp`, `.main`, `html`, `body`)은 **라이트 그레이(#F8F9FA)** 또는 **화이트(#FFFFFF)**로 고정한다.  
기존 어두운 그라데이션 배경 및 Glassmorphism 배경은 이 조항에 의해 폐기되며, CSS 후위 `!important`로 강제 적용된다.  
기본 텍스트(`p`, `li`, `span`)는 **짙은 차콜 그레이(#333333)**로 강제 지정하며, 어떠한 `text-shadow`도 허용하지 않는다.

### §1 CSS 구현 클래스
- `html, body` → `background: #FFFFFF`
- `.stApp`, `.main`, `.block-container` → `background: #F8F9FA`
- `p, li, span` → `color: #333333; text-shadow: none`

## 제7조 [프리미엄 그라데이션 박스 — Bright Cyan]

AI 핵심 요약 보드, 주요 컨테이너에 적용할 `.premium-gradient-box` CSS 클래스를 사용한다.  
박스 내부의 모든 텍스트 및 아이콘은 **반드시 흰색(#FFFFFF)**으로 처리하여 가독성을 보장한다.

### §1 CSS 속성
- `background: linear-gradient(135deg, #4facfe 0%, #00f2fe 100%) !important`
- `border-radius: 15px !important`
- `box-shadow: 0 8px 20px rgba(0, 192, 255, 0.2) !important`
- 내부 텍스트: `color: #FFFFFF !important`

### §2 사용 지침
HTML에서 `<div class="premium-gradient-box">` 형태로 사용하며, Streamlit `st.markdown(..., unsafe_allow_html=True)`로 렌더링한다.

## 제8조 [레드 얼럿 시스템]

경고, 부족한 보장 금액, 위험 상황 표시에는 `.red-alert-box` 및 `.red-text` 클래스를 사용한다.

### §1 CSS 속성
- `.red-alert-box`: `border: 1.5px solid #FF4B4B; background-color: rgba(255,75,75,0.05); border-radius: 8px`
- `.red-text`: `color: #FF4B4B`
- 박스 내부 텍스트: 모두 `#FF4B4B` 강제 적용

### §2 적용 대상
보장공백 경고, 실손 미가입 알림, 법령 위반 소지 표시, 계약 위험 경고 등 즉각 주의가 필요한 모든 UI 요소.

## 제9조 [공간 구획 및 경계]

모든 주요 페이지, 카드 블록, 익스팬더, 폼의 외곽 테두리는 **얇은 실선(`border: 1px solid #E0E0E0`)**으로 처리한다.  
섹션 간 가로 구분선은 CSS 마진으로 명확히 분리하며, 과도한 내부 그림자(box-shadow)는 사용을 자제한다.

### §1 CSS 타겟
- `[data-testid="stExpander"]`, `[data-testid="stForm"]` → `border: 1px solid #E0E0E0; border-radius: 10px`
- Expander summary → `border: 1px solid #E0E0E0; background: #FFFFFF`
- 세로 블록 간격: `margin-bottom: 6px`

## 제11조 [반응형 타이포그래피 — Fluid Typography]

기기 화면 너비에 따라 글자 크기가 자동 조절되도록 CSS `clamp()` 함수를 전면 적용한다.  
좁은 모바일 화면에서 긴 단어가 깨지지 않도록 `word-break: keep-all; overflow-wrap: break-word`를 전역 강제 적용한다.

### §1 폰트 크기 기준

| 요소 | CSS clamp() 값 |
|---|---|
| `p, li, span` (본문) | `clamp(13px, 1.2vw + 10px, 16px)` |
| `h1` (메인 타이틀) | `clamp(20px, 3vw + 12px, 28px)` |
| `h2` (섹션 타이틀) | `clamp(18px, 2.5vw + 10px, 24px)` |
| `h3` (카드 타이틀) | `clamp(16px, 2vw + 10px, 20px)` |
| Expander 내부 | `clamp(12px, 1vw + 10px, 14px)` |

### §2 전역 단어 줄바꿈
`*, *::before, *::after` → `word-break: keep-all; overflow-wrap: break-word`

## 제30조 [반응형 기기 최적화 (Device Adaptation)]

모든 UI(특히 스플래시 화면)는 '반응형 레이아웃'을 적용한다.  
접속 기기의 가로 너비를 실시간 감지하여, 핸드폰(Portrait)과 태블릿(Landscape) 모드에 맞는 최적의 이미지 비율을 자동 선택한다.  
스플래시 로고와 애니메이션은 화면 중앙 정렬을 원칙으로 하며, 화면 크기에 따라 로고 크기를 30%~50% 범위 내에서 유동적으로 조절한다.  
기기 종류에 상관없이 텍스트 가독성을 확보하기 위해, 최소 폰트 크기는 14px 이상을 유지한다.

## 제31조 [기기별 스플래시 이미지 분리 적용 (Dual Splash Strategy)]

모든 접속 기기에서 최적의 브랜드 경험을 제공하기 위해, 핸드폰과 태블릿용 스플래시 이미지를 별도로 운영한다.

### §1 모바일 최적화
가로 너비 **768px 미만** 환경에서는 세로형 전용 이미지(`splash_mobile.png`)를 호출하여 화면의 상하 여백을 최소화한다.  
실제 파일명: `Image_for_phone_splash_04c6295711.jpeg`

### §2 태블릿/PC 최적화
가로 너비 **768px 이상** 환경에서는 가로형 전용 이미지(`splash_tablet.png`)를 호출하여 광활한 가로 해상도를 100% 활용한다.  
실제 파일명: `Tablet_splash_screen_7b65c0fcd7.jpeg`

### §3 전환 부드러움
기기 회전 또는 크기 변경 시 이미지가 즉각 교체되되, 시각적 충격을 줄이기 위해 **0.5초의 페이드(Fade) 효과**를 강제한다.  
배경 레이어(`#gk-sp-bg`)의 `opacity`를 0 → 1로 전환하여 이미지 교체 충격을 최소화한다.

### §4 구현 우선순위
1. `window.innerWidth` 기준 실시간 감지 (DPR 보정 포함)
2. 768px 경계에서 이미지 분기 (`< 768` → mobile, `≥ 768` → tablet)
3. `resize` 이벤트 감지 → 경계 통과 시에만 이미지 교체 실행
4. 교체 시 `opacity: 0 → 1` fade 0.5s ease-in-out 적용

## 제32조 [CFP 기반 라이프사이클 및 의학경제학적 보장 분석 표준]

> AI는 고객의 경제적 가치를 산정할 때 반드시 이 조항의 공식과 논리를 최우선으로 적용한다.

### 제1항 [의학경제학적 가처분 소득 역산 원칙]

고객의 경제적 가치를 객관적으로 산출하기 위해 **당해 연도 전체 건강보험료율(2024년 기준 7.09%)**을 적용한 역산법을 최우선 지표로 삼는다.

**추정 월 소득 산출 공식:**
```
추정 월 소득 = 납입 건강보험료 ÷ 0.0709
```

**일일 경제적 가치(필요 일당) 산출 공식:**
```
일일 필요 일당 = 추정 월 소득 ÷ 30일
```

예시: 건보료 25만 원 납입 시 → 월 소득 약 352만 원 → 일일 가치 약 12만 원

### 제2항 [상해후유장해 소득 대체 로직 — 2년 원칙]

상해로 인한 10~15%의 영구장해 발생 시, 신체적 손실을 넘어 **'시간의 상실'**을 보전하는 것을 목적으로 한다.

- **소득 대체 기간:** '치료 및 적응기(1년)' + '재취업/직장적응기(1년)' = 총 **2년**의 소득 공백 전제
- **적정 가입 금액:** 10% 장해 시 2년치 소득을 온전히 보전할 수 있도록 역산 소득 기반 설계, **최소 3억~5억 원 필수 방어선** 권고

### 제3항 [의학적 근거에 따른 질병 및 간병 설계]

질병의 판정 기간과 현대 의학의 치료비 현실을 반영하여 실질적 비용 공백을 메운다.

| 질환 | 기준 기간 | 설계 기준 |
|------|----------|----------|
| **중풍(뇌졸중)** | 국가 영구장해 판정·요양 지원까지 최소 **18개월** | 18개월치 이상 소득 대체 + 간병 비용 |
| **치매** | 확진 후 장기요양 신청 가능 경과 **6개월** | 최소 6~8개월치 간병 비용 |
| **암 진단비** | 조기암(5천~1억)과 별개 | 진행성/전이암: 표적·면역항암제 비용 반영, **최소 1억~3억 원** 권장 |

### 제4항 [CFP 기준 라이프사이클 연금 설계]

은퇴 후 삶의 질을 결정하는 소득 대체율을 CFP(국제공인재무설계사) 표준에 따라 구성한다.

**3층 연금 구조:**
- 국민연금(40~45%) + 퇴직연금(10%)의 한계를 인정
- **개인연금이 현 소득의 40~50%를 보전**할 수 있도록 설계

### 제5항 [지능형 담보 매칭 및 보전 전략 — Gap Filling]

금융감독원(FSS)의 합산 한도 제한으로 인해 발생하는 보장 공백을 전략적으로 해결한다.

- **한도 초과 대응:** 계산된 일일 필요 일당이 금감원 한도(상해 7만 / 질병 10만 등)를 초과할 경우 차액을 즉시 산출
- **담보 복합 설계:** 차액을 보전하기 위해 상해/질병 수술비, 1~5종 수술비, N대 질병 수술비 등을 조합하여 입원 시 실제 수령액이 고객의 일일 경제적 가치에 수렴하도록 최적화

### 제6항 [전문가 브리핑 및 리포트 작성 표준]

분석 결과 제시 시 고객의 신뢰를 확보하기 위해 다음의 용어와 논리를 강제한다.

- **핵심 용어 사용:** 단순 금액 설명 대신 **'의학경제학적 휴업손해 보전'** 용어 사용
- **논리적 연결:** "왜 수술비 담보가 필요한가?"에 대해 건강보험료 역산 결과(고객의 실질 가치)와 연동하여 설명 — 설계사 개인의 주관이 아닌 **데이터 기반 객관적 진단**임을 명시
- **브리핑 표준 문구:**
  > "고객님의 건강보험료 기반 일일 경제적 가치는 OO만 원이나, 규정상 일당은 OO만 원까지만 가입 가능하므로 부족분 OO만 원을 수술비 담보로 대체 보완하였습니다."

## 제34조 [시스템 자율 유지보수 및 법정 요율 최신화 (Autonomous Maintenance)]

> 이 조항은 앱이 시간이 지나도 '낡은 지식'을 전달하지 않도록 방어하는 **'디지털 방부제'** 역할을 한다.

### 제1항 [법정 요율 정기 모니터링]

시스템은 매년 1회(매년 1월 1일~5일 사이) 보건복지부 및 국민건강보험공단의 공시 자료를 탐색한다.

모니터링 대상:
- 당해 연도 전체 **건강보험료율** (2024년 기준 7.09%)
- **장기요양보험료율**
- **산재보험 요율** 등 역산 알고리즘에 사용되는 모든 법정 수치

### 제2항 [알고리즘 자율 보정]

- 요율 변동이 확인될 경우, 제32조에 명시된 역산 공식의 **분모(Denominator) 수치**를 즉각 자동 수정한다.
- 수정된 수치는 즉시 **컨설팅 슬라이더 및 시뮬레이터**에 반영된다.
- 코드 내 `0.0709` 하드코딩 상수는 **`_CURRENT_NHIS_RATE`** 동적 변수로 관리하며, Supabase `system_config` 테이블에 저장·로드한다.

### 제3항 [업데이트 로그 및 투명성 확보]

- 요율이 변경될 때마다 관리자(설계사)에게 **인앱 알림 배너**를 발송한다.
- 가이딩 프로토콜 문서 하단 `[Update Log]` 섹션에 자동 기록한다.
- 고객 리포트 하단에 다음 문구를 자동 갱신한다:
  > "본 분석은 {연도}년 {월}월 공시된 최신 요율(건강보험료율 {요율}%)을 기준으로 작성되었습니다."

### 제4항 [지식의 동적 확장]

- 요율 외에도 **금감원 지침 변경** 및 **의학적 신기술**(신규 고가 항암제 등) 출시 정보를 정기적으로 학습한다.
- 제32조의 보장 기준(암 진단비 등)을 상향 제안할 수 있도록 시스템을 최적화한다.
- 가이딩 프로토콜 제28조(동적 지식 확장 시스템)와 연계하여 시스템 신뢰성을 유지한다.

## 제35조 [치매 및 퇴행성 질환 전문 보장 및 자율 업데이트 원칙]

알츠하이머 신약(레켐비·도나네맙 등)의 임상 효능은 **초기 단계(CDR 0.5~1)**에 집중된다.

## 제37조 [실용 보안 진입 원칙 (Practical Security Entry — 2-Step Auth Only)]

> 시스템은 **'가문 안보'**의 중요성에 따라 보안과 상담 연속성의 균형을 최우선으로 한다.  
> **(2026년 개정)** 생체인식·패턴·PIN 인증은 완전 폐기되었으며, **2단계 인증(ID/PW → 6자리 OTP)** 이 유일한 표준 진입 절차이다.

### 제1항 [인증 단순화 — 2단계 고정]

- 인증 절차는 **Phase A(아이디/비밀번호)** → **Phase B(6자리 OTP)** 두 단계로 완전히 고정한다.
- Phase SETUP(생체/패턴/PIN 설정) 및 Phase C(생체/패턴/PIN 로그인)는 **코드베이스에서 완전히 제거**되었다.
- 생체인식(`지문/Face ID`), 패턴 잠금, PIN 인증은 본 시스템에서 **영구 폐기**한다.

### 제2항 [OTP 즉시 진입 보장]

- OTP 6자리 인증 성공 즉시 `_do_final_login()`을 호출하고 `st.rerun()`으로 메인 대시보드에 진입한다.
- OTP 인증 성공 후 **추가 단계, 추가 UI, 추가 확인 없이** 즉시 메인 화면이 렌더링되어야 한다.
- `_lp` 세션 키가 `"SETUP"` 또는 `"C"`인 경우 즉시 `"A"`로 리셋한다(잔존 세션 보호).

### 제3항 [UX 최적화 — OTP 입력창 대형화]

- OTP 6자리 입력창은 **가이딩 프로토콜 제36조(돋보기 모드)** 원칙에 따라 대형 폰트로 배치한다.
- 입력 글자 크기: **최소 2rem 이상**, 중앙 정렬, 굵은 자간(letter-spacing: 0.3em 이상).
- 설계사가 현장에서 번호를 빠르고 정확하게 입력할 수 있도록 입력 오류를 최소화한다.

### 제4항 [세션 연장 — 상담 연속성 보장]

- 로그인 성공 후 상담 세션은 **2시간(7,200초)** 동안 유지된다.
- 비활동 세션은 2시간 경과 시 자동 만료된다.
- 인증 대기 화면 상단에 **"🛡️ 가문 안보를 위한 2단계 보안 가동 중"** 문구를 필수 표시한다.

### §구현 핵심 상수 (app.py)

| 상수/변수명 | 역할 |
|------------|------|
| `SESSION_TTL` | 세션 유지 시간 (초, 기본 7200 = 2시간) |
| `_session_checkin()` | 세션 체크인 및 만료 처리 함수 |
| `_get_session_remaining()` | 잔여 세션 시간(초) 반환 |

| `_do_final_login()` | OTP 인증 성공 후 즉시 호출되는 최종 로그인 처리 함수 |

## 제38조 [전략적 경량화 및 지능형 배치 처리 원칙 (Strategic Lightweighting & Intelligent Batch Protocol)]

> 앱의 **민첩성**을 극대화하고, 지식의 깊이는 **야간 학습**을 통해 채운다.  
> 앱은 낮에는 설계사의 상담을 돕고, 밤에는 스스로 공부하여 다음 날 더 나은 답변을 준비한다.

### 제1항 [시각적 부하 제거 — UI Lightweighting]

- 페이지 전환 및 블록 이동 시 발생하는 **모든 돋보기 애니메이션, 암전 효과, 복잡한 렌더링 프로세스(CSS filters, JS heavy effects)를 소스 코드에서 완전히 삭제**한다.
- 모든 화면 전환은 **페이드인(Fade-in) 또는 즉시 표시(Instant Display)** 로 대체하여 체감 속도를 제로화한다.
- 화면 전체를 어둡게 만드는 다크 오버레이(`gk-loading-overlay`)와 스피너는 폐기하며, `gk-show` 클래스 부여를 중단한다 (하위 호환 DOM 유지).
- GCS 데이터 로드 대기 중에는 **해당 블록의 형태를 미리 보여주는 스켈레톤 UI를 최우선으로 렌더링**한다.
- 체감 대기 시간 목표: **0.5초 이내**.
- 표준 구현 클래스: `.gk-skeleton` (shimmer loader), `transition: opacity 0.25s ease-in-out`.

### 제2항 [핵심 단어 추출 및 저장 — Daytime Keyword Logging]

- 상담 중 발생하는 질문에 대해 **전체 텍스트 분석 대신, 명사 위주의 핵심 키워드(Key Words)만을 추출**하여 가벼운 JSON 형태로 별도의 **야간 학습 대기열(Pending Queue)** 에 기록한다.
  - 예: `"레켐비 부작용"`, `"산정특례 갱신"` 등 핵심 단어 단위 저장.
- **낮 시간대에는 분석을 멈추고 기록만 수행**한다 — 실시간 무거운 처리는 야간으로 위임한다.
- Pending Queue 저장 항목: `핵심 키워드`, `질문 원문`, `발생 시각(ISO 8601)`, `컨텍스트(탭/기능명)`, `해결 여부`.
- 저장소: `st.session_state["_pending_queue"]` (세션 내 누적) → Supabase `pending_wisdom` 테이블 비동기 저장.
- 기록 트리거 조건: LLM 응답에 `검색 필요`, `확인 후 안내`, `정보 없음`, `추가 확인`, `정확한 정보가 없`, `확인이 필요` 패턴이 포함된 경우.

### 제3항 [야간 자율 지능형 캐싱 — Night-time Intelligence]

- 앱 사용량이 최저인 **심야 시간대(02:00~04:00 KST)** 에 시스템은 스스로 저장된 키워드를 분석하여 심층 자료를 검색한다.
- 야간 자율 프로세스 순서:
  1. Supabase `pending_wisdom` 로드 → `resolved=False` 항목 최대 20건 추출
  2. Google Search 및 LLM(Gemini)을 통해 정교한 답변과 시각 자료 생성
  3. 결과물을 **'상담 준비 패키지'** 형태로 GCS `consulting_cache/{md5}.json` 저장 → 다음 날 즉시 호출 가능
  4. GCS `wisdom_cache.json` 갱신 (새 인사이트 목록 누적)
  5. Supabase `pending_wisdom.resolved = True` 플래그 갱신
- 야간 프로세스 실행 코드에는 `# [제38조 §3] 심야 전용 — 02:00~04:00 KST` 주석을 필수 표기한다.
- **현재 구현 수준**: `_art38_night_worker()` 완성 (Supabase 연동 + Gemini+GoogleSearch 호출 + GCS 저장).

### 제4항 [상담 브리핑 준비 — Morning Insight Banner]

- 야간 학습이 완료된 경우, 설계사님이 앱을 구동할 때 **'밤새 정리된 새로운 인사이트'** 알림 배너를 최상단에 표시한다.
- 배너 표시 조건: `wisdom_cache.json`에 새 항목이 존재하며 `shown: false`인 경우.
- 캐시 메모리 전진 배치: 새 인사이트를 `st.session_state["_wisdom_ready"]`에 로드하여 해당 질문이 다시 들어올 때 즉시 응답 (**Cache-First 전략**: 로컬 세션 캐시 → GCS 순).
- 배너 확인 후 `shown: true` 플래그 갱신 → 중복 표시 방지.

### 제5항 [성능 개선 보고 의무 — Performance Report]

- 제38조 조치 적용 후, **앱의 초기 로딩 속도 및 페이지 전환 속도 개선 수치를 수치로 보고**할 의무가 있다.
- 보고 기준: 조치 전/후 체감 대기 시간(ms) 및 GCS 호출 횟수 비교.
- 캐시 히트율(consulting_cache hit rate)을 지속적으로 모니터링하여 야간 학습 효율을 평가한다.

### §구현 핵심 상수 및 함수 (app.py)

| 항목 | 역할 |

## 제39조 [무결성 순차 전환 프로토콜 (Sequential Zero-Delay Transition Protocol)]
**(2026년 최종 개정)** 앱 진입 시 '기다림 없는 전문성'을 제공하기 위한 3단계 순차 구동 체계.

> **진입 구조**: `[기기 감지형 스플래시 랜딩]` → `[안보 서비스 시작하기]` → `[로그인]` → `[사이드바 Shell]` → `[본앱]`  
> 구 스플래시 컴포넌트(`st.image`, `gk-sp-hide-style` 등) **완전 폐기** — 재생성 금지.  
> 돋보기(`st.spinner`, `gk-spinner`) **영구 제거** — 모든 전환은 스켈레톤 UI로 대체.

### 제1항 [기기 감지형 하이브리드 랜딩 — Device-Aware Hybrid Landing]

- 앱 구동 즉시 `_s39_render_landing_page()`가 **풀스크린 오버레이(`position:fixed, z-index:9999`)**로 랜딩 페이지를 표시한다.
- **기기 해상도 감지**에 따라 배경 이미지를 분기한다:
  - **Mobile (세로 / portrait)**: `assets/splash_goldkey.webp` (b64: `splash_goldkey.b64`)
  - **Tablet (가로 / landscape, ≥600px)**: `assets/Tablet_splash_screen_7b65c0fcd7.jpeg` (b64: `tablet_splash.b64`)
  - CSS `@media (min-aspect-ratio:3/4) and (min-width:600px)` 미디어 쿼리로 분기.
- **이미지 피팅**: `background-size:cover; background-position:center;` — 모든 해상도에서 왜곡 없는 풀스크린.
- 하단 그라디언트 오버레이(`::before`, `rgba(3,9,20,0.82)`)로 텍스트 가독성 확보.
- 랜딩 페이지는 **세션당 1회** 노출. 세션 플래그: `_lp_landing` (`True` 시 건너뜀).

### 제2항 [백그라운드 예비 로딩 — Background Pre-fetching]

- 랜딩 화면 노출 **즉시** 클라이언트 사이드 JS 블록 실행:
  1. **LocalStorage 세션 캐시 읽기** (`gk_session_cache`) — 기존 세션 유효 여부 즉시 파악.
  2. **폰트/CSS prefetch 링크 주입** — Noto Sans KR 폰트를 미리 워밍하여 로그인 화면 즉시 렌더 보장.
  3. **WebSocket keepalive 타임스탬프** (`window._gk_landing_ts`) — 재연결 지연 방지.
  4. **MutationObserver** — 버튼이 DOM에 마운트되는 즉시 감지하여 fixed 래퍼(`#gk-start-btn-wrap`)로 이동.

### 제3항 [동시성 교차 전환 — Simultaneous Seamless Transition]

- `[🚀 안보 서비스 시작하기]` 버튼 클릭 시 **클라이언트·서버 동시 실행**:
  - **클라이언트**: `#gk-landing-root`에 `.gk-lp-exit` 클래스 추가 → `opacity:0` (150ms 전환) → `display:none`.
  - **서버**: `_lp_landing=True`, `current_tab="home"` 세팅 → `st.rerun()`.
- 랜딩 제거와 사이드바 Shell 렌더가 **동시에** 이루어져 빈 화면·돋보기 노출 Zero.
- **목표**: 버튼 클릭 → 사이드바 노출까지 **0.3초 이내**.

### 제4항 [사이드바 Shell-First 렌더링 유지]

- 로그인 후 `_s39_sidebar_shell_html()` 텍스트 메뉴 뼈대를 즉시 노출.
- 프로필 이미지·통계는 **백그라운드 스레드** 비동기 주입.
- 세션 캐시 저장: `user_id`, `user_name`, `user_role`, `last_login`. 유효기간: **24시간**.

## 제40조 [0.5초 무결성 유지 및 자율 최적화 원칙 (Sub-Second Integrity & Auto-Optimization Protocol)]
> 시스템이 스스로 성능을 진단하고 기준 미달 시 즉각 보정하도록 강제한다. 모든 인터랙션의 시각적 피드백은 **500ms 이내**에 완료되어야 한다.

### 제1항 [0.5초 임계치 준수 — 0.5s Threshold]

- 모든 페이지 전환, 데이터 출력, 버튼 인터랙션은 실행 후 **500ms 이내**에 시각적 피드백을 완료해야 한다.
- 임계치를 초과하는 요소는 **차단 요소(Blocking Element)**로 분류하며, 즉각 비동기 처리 또는 Skeleton UI로 대체한다.
- Streamlit `st.stop()` 전 모든 blocking I/O(GCS, Supabase, 파일 탐색)는 **백그라운드 스레드**에서만 실행한다.

### 제2항 [병목 현상 자율 탐지 — Auto Bottleneck Detection]

- 앱 구동 시마다 Critical Rendering Path를 순회하여 다음 항목을 자동 식별한다:
  1. 동기 GCS/Supabase 호출 (화면 차단형 `await`)
  2. 500ms 초과 연산 (복잡한 JS 리렌더링, 대용량 HTML 문자열 생성)
  3. 미사용 라이브러리 임포트 및 중복 CSS 선언
- 탐지된 항목은 `_s40_perf_log` 세션 키에 누적하고, 야간 배치(`_art38_night_worker`) 실행 시 자동 리포트를 생성한다.

### 제3항 [즉각적 성능 수술 — Optimization Surgery]

- **차단 제거**: 화면 표시 전 데이터를 기다리는 모든 동기 호출을 `threading.Thread(daemon=True)`로 교체하고, 데이터 도착 전까지 `.gk-skeleton` shimmer UI를 노출한다.
- **캐싱 강화**: 역산 수치·가이드라인 텍스트는 `st.session_state` + JS `localStorage`에 이중 저장하여, 재구동 시 GCS 호출 없이 **0.1초 이내** 출력한다.
- **번들 최적화**: 이미지는 `loading="lazy"` + WebP/base64 data-URI 우선 사용, 아이콘은 SVG 인라인 또는 이모지 대체.

### 제4항 [0.5초 모니터링 시스템 — Performance Watchdog]

- 각 탭 전환 시점에 JS `performance.now()` 타이머를 주입하여 렌더 완료 시간을 측정한다.
- **500ms 초과 시** `console.warn('[GK-PERF]')` 경고 + 세션 로그(`_s40_perf_log`) 기록.
- 야간 배치 실행 시 `_s40_perf_log` 분석 → 상위 3개 병목 탭을 `_s40_slow_tabs` 세션 키에 저장.

### §구현 핵심 상수 및 함수 (app.py)

| 항목 | 역할 |
|------|------|
| `_S40_THRESHOLD_MS` | 성능 임계치 (기본값: `500` ms) |
| `_S40_PERF_LOG_KEY` | 성능 로그 세션 키 (`"_s40_perf_log"`) |
| `_s40_perf_watchdog_js()` | 탭 전환 시 타이밍 측정 + 500ms 초과 경고 JS 반환 |
| `_s40_record_blocking()` | 차단 요소 발견 시 세션 로그에 기록 |
| `_s40_night_report()` | 야간 배치용 병목 분석 리포트 생성 |

## 제41조 [내부 식별자 최적화 및 메타데이터 분리 원칙 (Internal ID Optimization & Metadata Separation Protocol)]

> 기계가 읽는 부분은 최대한 가볍게(English/ID), 사람이 읽는 부분은 친절하게(Korean) 분리하여 속도를 극대화한다. 앱 내부 로직에서 한글 문자열 비교를 전면 금지하고, 영문 상수 기반 식별자 체계로 전환한다.

### §1 식별자 영문 표준화 (Internal ID Standardization)

- 모든 가이딩 프로토콜 조항은 `GP_ID_NN` 형식의 영문 상수로 참조한다.
- 앱 내부 조건문·루프·API 호출에서 한글 문자열 비교(`if x == "제40조..."`)를 **전면 금지**한다.
- 런타임 텍스트 파싱 시간을 0에 수렴시키기 위해 정수·문자열 상수(`GP_ID_*`)를 사용한다.

### §2 다국어 매핑 테이블 (GP_CONSTITUTION + GP_UI_DICT)

- `GP_CONSTITUTION` 딕셔너리: 영문 키 → 프로토콜 메타데이터(조항 번호, 키워드) 매핑.
- `GP_UI_DICT` 딕셔너리: 영문 GP_ID → 한글 표시명 매핑 (UI 렌더링 전용).
- 내부 엔진(Backend/Logic)은 오직 `GP_ID_*` 상수만 취급한다.

### §3 바이너리 친화적 코딩 원칙

- 상수 비교: `if gp_id == GP_ID_40` (O) / `if label == "제40조"` (X)
- 새 조항 추가 시 `GP_CONSTITUTION`과 `GP_UI_DICT`에 동시 등록 의무.
- 조항 참조 로그는 `GP_ID` 키로만 기록하며, 한글 레이블은 출력 직전 `GP_UI_DICT`에서 변환.

### §구현 핵심 상수 및 함수 (app.py)

| 항목 | 역할 |
|------|------|
| `GP_CONSTITUTION` | GP_ID → 조항 메타데이터 딕셔너리 (`article`, `keyword`, `threshold` 등) |

## 제70조 [상급종합병원 실질 위협 산출 및 근거 명기 원칙 (Big5 Master Cost & Evidence Protocol)]

> 본 조항은 일반적인 국가 통계의 착시(평균의 함정)를 제거하고, 대한민국 의료의 정점인 서울 5대 병원(서울아산·삼성서울·서울대·세브란스·성모) 기준의 비급여 실부담액을 산출하며, 반드시 그 근거를 지식 베이스에 기록함을 목적으로 한다.

### §1 [비급여 프리미엄 가중치 — Big5 Premium Factor]

| 항목 | 일반 병원 (국가평균) | 서울 5대병원 (마스터형) | 근거 출처 |
|------|-----------------|---------------------|---------|
| 로봇수술(다빈치) 비급여 | 800~1,000만원 | 1,500~2,500만원 | 서울대·세브란스 비급여 고시 2025 |
| 1인실 병실료 (7일) | 약 140만원 | 350~420만원 | 아산·삼성 1인실 평균 수가표 2024 |
| 표적/면역항암제 (연간) | 급여 적용 시 일부 | 6,000~1억원 | 심평원 비급여 약제 통계 2024 |

- **로봇수술(다빈치)**: 전립선암 1,500~2,500만원, 위·대장암 800~1,500만원 (일반 종합병원 대비 약 1.5배)
- **상급병실료(1인실)**: 일 40~60만원 (국민건강보험 미적용 항목, 심평원 비급여공시 2024)
- **표적/면역항암제**: 비급여 투여 시 1주기당 수백만원, 연간 최대 1억원 상회
  - 키트루다(폐암): 3주 1회 × 연 17회, 1회 약 390만원 = 연간 약 7,000만원 (서울대보라매병원 2024 비급여공시)
  - 옵디보: 2주 1회 약 340만원 (바이오스펙테이터 2024)
  - 허셉틴(유방암 HER2+): 3주 1회 × 18회, 1회 약 130만원 = 연간 약 2,400만원 (국립암센터 2023)
- **Big5 가중치**: 암 1.7배, 뇌 1.5배, 심 1.4배, 만성 1.3배 (비급여 비중 60~80% 반영)

#### §1-A [암종별 로봇수술 비급여 상세 수가 — Cancer-Specific Robot Surgery Cost]

| 암종 | 국가평균(종합병원) | 서울 5대병원 | 대표 병원 | 근거 출처 |
|------|----------------|------------|---------|---------|
| 전립선암 | 1,000~1,200만원 | 1,500~2,500만원 | 서울대·세브란스 | 비급여 고시 2025.02 |
| 위암 | 800~1,000만원 | 1,000~1,500만원 | 서울아산·삼성 | 보건복지부 비급여보고제도 2025 |
| 대장암 | 800~1,000만원 | 1,000~1,500만원 | 서울아산·서울대 | 보건복지부 비급여보고제도 2025 |
| 간암 | 1,000~1,200만원 | 1,200~1,800만원 | 삼성·서울아산 | 심평원 비급여공시 2024 |
| 유방암 | 600~900만원 | 800~1,200만원 | 세브란스·성모 | 심평원 비급여공시 2024 |
| 폐암 | 해당없음(개흉) | 해당없음 | — | 로봇보조 흉강경 일부 적용 |

- **암종별 로봇수술 발동 기준**: `big5_robot_range != "해당없음"` 인 경우에만 대비 테이블에 로봇수술 행 노출
- **일반 종합병원 대비 Big5 배율**: 전립선 1.5~2.1배, 위·대장 1.2~1.5배, 간 1.2~1.5배

### §2 [지식 베이스 근거 명기 의무 — Mandatory Evidence Tagging]

모든 비급여 데이터는 `_GP68_DISEASE_DB` 내 `big5_source` 필드에 다음 출처 중 하나 이상을 반드시 매핑하여 저장한다.

- **[병원 공시]**: 각 병원 홈페이지 '비급여 진료비 안내' 자료
- **[심평원]**: 건강보험심사평가원 비급여 진료비용 공개 데이터
- **[협회/기관]**: 국립암센터 암정보센터 및 대형 보험사 통계 리포트

각 질병 데이터 키 구조:
```python
"big5_robot_cost":    int,   # 로봇수술 비급여(만원) — 병원 공시 기준
"big5_immuno_annual": int,   # 면역항암제 비급여(만원/년) — 심평원·병원 공시

"big5_source":        str,   # 출처 문자열 (institution_name + update_date)
"big5_source_link":   str,   # 출처 URL (심평원 공시 페이지 등)
```

### §3 [마스터형 급여폭탄 멘트 자동 생성 조건 — Answer Generation Conditions]

**발동 조건** (AND 조건, 모두 충족 시에만 멘트 생성):
1. `_gp70_big5 == True` — 5대병원 마스터형 선택
2. `(_robot_cost + _immuno_ann) > 0` — 로봇수술 또는 면역항암제 비급여 비용이 존재

**멘트 구성 요소**:
- ① 로봇수술 비급여(`big5_robot_cost`) + 면역항암제 비급여(`big5_immuno_annual`) 합계 → `_fmt_man()` 변환 강조
- ② 면역항암제 산출근거 병명별 자동 분기 (`_immuno_basis` 변수):
  - 폐암: 키트루다(Pembrolizumab) 3주 1회 × 연 17회, 1회 약 390만원 — 서울대보라매병원 2024 비급여 공시

  - 대장암·간암: 면역항암제 비급여 적용 기준 — 보건복지부 비급여보고제도 2025

- ④ 마스터형 선택 시 **시각적 대비 테이블** 자동 노출 (§4 참조)

- 산출근거 주석: `color:#374151` (진회색), `font-size:0.78rem`

### §4 [시각적 대비 로직 — Visual Contrast Table]

- 테이블 구조: `항목명 | 국가 평균 | 5대병원 마스터형 | 근거` 4열

- 출처 링크 클릭 시 심평원 비급여 공시 페이지로 연결

### §5 [위협 지수 산출 공식 — Threat Index Formula]

$$Total\_Threat = (NonBenefit\_Cost \times Big5\_Factor) + (Monthly\_Income \times Recovery\_Months)$$

- `NonBenefit_Cost`: `big5_robot_cost + big5_immuno_annual + (big5_room_per_day × hosp_days_avg)`
- `Big5_Factor`: 암=1.7, 뇌=1.5, 심=1.4, 만성=1.3
- `Monthly_Income`: 건보료 역산 월소득 (제32조 §1 공식)
- `Recovery_Months`: `work_loss_months` (질병별 평균 휴직 기간)
- 급여 본인부담금은 비급여 대비 미미한 수준으로 간주, 계산 비중 최소화

**구현 변수**:
| 변수 | 역할 |
|------|------|
| `_robot_cost` | `_loss.get("robot_cost", 0)` |
| `_immuno_ann` | `_loss.get("immuno_annual", 0)` |
| `_room_extra` | `_loss.get("room_extra", 0)` |
| `_immuno_basis` | 병명별 면역항암제 근거 문구 |
| `_master_intro` | 최종 HTML 멘트 (조건 불충족 시 빈 문자열) |
| `_contrast_table_html` | 국가 평균 vs 5대병원 대비 테이블 HTML |

**Staleness 경고 표시 규칙**:

- 경고 배지 CSS: `.gk-evidence-badge` + `background:#D97706;color:#fff`
- Constitution.md Update Log에 갱신 이력을 반드시 기록

## 제72조 [담보별 실질 가용액 산출 및 합산 금지 원칙 (Anti-Aggregation Coverage Valuation)]

> 본 조항은 성격이 다른 표적항암 담보를 단순 합산하여 고객에게 착시를 주는 행위를 금지하며, 실제 치료 상황별로 가용 가능한 금액을 분리 산출한다.

### §1 [카테고리별 분리 산출 — Anti-Aggregation]

표적항암 관련 담보는 치료 시나리오별로 분리하여 산출하며, 합산 총액을 전면에 내세우지 않는다.

| 시나리오 | 포함 키워드 | 표시 방식 |
|---------|-----------|---------|
| 약물치료 시나리오 | `표적항암약물`, `항암약물허가`, `항암약물`, `표적항암` | `💊 약물치료 시 가용: X만원` |
| 방사선치료 시나리오 | `항암방사선`, `방사선약물`, `방사선치료` | `☢️ 방사선치료 시 가용: Y만원` |
| 면역항암 시나리오 | `면역항암`, `면역치료` | `🧬 면역항암 시 가용: Z만원` |

- UI 표현 금지: `"총 표적항암 보장액: A+B+C = X원"` 형식의 합산 총액 전면 노출
- UI 표현 필수: `"치료 방식별 가용 금액"` 용어 사용

### §2 [상호 배타적 담보 처리 — Mutual Exclusivity Rule]

동일 치료 에피소드에서 약물치료 담보와 방사선치료 담보는 중복 수령 불가하므로, 보장 자산 산출 시 **최대값 1개만** 유효 자산으로 인정한다.

```python
_amt_drug_excl = max(_amt_drug, _amt_rad)   # 중복 합산 금지
```

### §3 [시각적 분리 산출 배너 — GP-72 UI 발동 조건]

**발동 조건**: 암 카테고리 질병 선택 AND `_sc_drug | _sc_rad | _sc_immuno | _sc_major` 중 1개 이상 존재

**배너 구성**:
- `[GP-72] 치료 방식별 가용 금액 (합산 착시 제거)` 헤더
- 시나리오별 행 개별 표시 (약물 / 방사선 / 면역항암)
- 약물·방사선 동시 존재 시 `※ 약물·방사선 중 실제 선택 치료 1가지만 수령 가능` 주의문 노출
- 팩트 폭격 문구: `"실제 약물 치료를 받으실 때 쓸 수 있는 돈은 X만원뿐입니다. 숫자의 착시에 속으시면 안 됩니다."`

**구현 변수**:
| 변수 | 역할 |
|------|------|
| `_sc_drug` | 약물치료 담보 리스트 |
| `_sc_rad` | 방사선치료 담보 리스트 |
| `_sc_immuno` | 면역항암 담보 리스트 |
| `_sc_major` | 암주요치료비 담보 리스트 |
| `_amt_drug_excl` | 상호배타 처리 후 약물·방사선 최대값 |

## 제73조 [연간 반복 담보의 5년 한도 보장 자산 평가 원칙 (5-Year Asset Cap for Recurring Benefits)]

> 본 조항은 암 주요치료비 등 연간 반복 지급 담보의 자산 가치를 현실적인 치료 주기인 5년으로 제한하여 평가한다.

### §1 [5년 캡(Cap) 적용 — 5-Year Asset Ceiling]

암 주요치료비(`암주요치료비`, `주요치료비`, `암주요` 키워드 담보)의 보장 자산 계산 시:

- **확정 안보 자산**: 총 보장액 × 0.5 (10년 약관 기준 5년치)
- **희망 보장 영역**: 총 보장액 × 0.5 (6~10년차, 메인 안보 지수 미반영)

```python
_amt_major_5yr   = _amt_major_raw * 0.5   # 확정 안보 자산 (1~5년)
_amt_major_bonus = _amt_major_raw * 0.5   # 희망 보장 영역 (6~10년)
```

### §2 [시각적 구분 표시 — Color-Coded Asset Classification]

| 구간 | 표시 방식 | 색상 |
|------|---------|------|
| 1~5년차 (확정) | `⭐ 암주요치료비 [5년 확정]` | `#16a34a` (초록) |
| 6~10년차 (희망) | `📦 암주요치료비 [6~10년 희망]` | `#94a3b8` (회색) |

### §3 [팩트 폭격 문구 — Bot Scripting]

봇이 암주요치료비 분석 시 반드시 발화:

> "주요치료비 10년 보장도 실제로는 5년 집중 치료기(X만원)를 기준으로 준비하는 것이 가장 정교한 안보 전략입니다."

### §4 [총 가용 보장 자산 산출 공식]

```
총 가용 보장 자산 = max(약물치료, 방사선치료) + 면역항암 + 암주요5년
```

- `_gp71_matched_amt` 변수에 저장
- 5,000만원 미달 시 `⚠️ 첨단 치료 접근 불가능 상태` 경고 발동 (GP-71 §3 연동)

## 제51.2조 [비회원모드 한정적 기기 식별 원칙 (Non-Member Device Gating)] — 개정

본 조항은 비회원(Non-member)의 무분별한 사용을 억제하고 정회원 전환을 유도하기 위해 기기 지문을 '일회성 식별자'로만 활용한다.

### §1 [식별 범위 한정]

기기 지문(Fingerprint_ID)은 오직 비회원모드의 **접속 횟수 카운팅** 및 **일일 접속 제한 확인**용으로만 사용한다. 계정 복구 및 본인 인증 수단으로의 활용을 엄격히 금지한다.

### §2 [1-10 규정 (1 Day – 10 Times)]

- **일일 제한**: 동일 기기 지문 기준, 1일 1회만 비회원 접속을 허용한다.
- **누적 제한**: 기기당 총 10회까지만 비회원 로그인을 허용한다. 10회 소진 후에는 정회원 가입 화면으로 강제 연결한다.

### §3 [기술적 한계 명시]

브라우저 캐시 및 로컬 스토리지 초기화 시 식별 데이터가 소멸됨을 인정하며, 시스템은 이를 '신규 기기'로 간주하여 카운팅을 1회부터 재시작한다.

### §4 [시각적 알림 의무]

비회원 화면 상단에 "남은 체험 횟수: N회" 또는 "오늘의 체험 완료" 배너를 명확히 표시하여 사용자가 제한을 인지하게 한다.

### §5 [구현 위치]

- LocalStorage 키: `gk_guest_fp` (Fingerprint_ID), `gk_guest_visits` (`{total_visits, last_visit_date}`)
- 차단 로직: JS 클라이언트 측 1차 검증 + Streamlit 서버 세션 2차 검증 병행
- 비회원 버튼 클릭 시 JS 인터셉트 → 조건 분기 → 허용/차단 판정

## 제75조 [기기 지문 기반 본인 신뢰도 점수 운영 원칙 (Device Trust Score)]

기기 지문을 '신분증'이 아닌 **신뢰 지표**로 활용하는 프로토콜을 규정한다.

### §1 [신뢰도 가산점 (Trust Score Boost)]

비번 찾기/변경 요청 시, 현재 기기 지문이 기존 로그인 기록(Known_Devices)과 일치하면 **신뢰도 +50점**을 부여한다. 이 경우 간편 인증(이메일 확인 수준)으로 절차를 간소화한다.

### §2 [이상 징후 감지 (Anomaly Detection)]

비번 찾기를 시도하는 기기 지문이 Known_Devices에 없는 신규 값이면 **위험도 상승**으로 간주하고, 강화된 인증(추가 본인 확인 또는 관리자 직접 확인)을 요구한다.

### §3 [환경적 증거 보존 (Environmental Evidence)]

기기 지문과 더불어 최근 접속 시각, 주력 상담 분야를 조합하여 '본인 소유 여부'를 판정하는 입체적 근거로 활용한다.

### §4 [로그인 기기 이력 저장 (Device History Logging)]

- 로그인 성공 시 Fingerprint_ID를 `gk_device_history` 테이블(Supabase)에 저장한다.
- 저장 필드: `user_name`, `fp_id`, `last_seen`, `user_agent_hint`
- 최대 5개 기기 이력 보존 (초과 시 가장 오래된 항목 삭제)

### §5 [관리자 복구 지원 도구]

관리자 패널에 특정 회원의 Known_Devices 목록과 현재 요청 기기의 일치 여부를 판정하여 표시한다. 이를 통해 관리자의 복구 승인 판단을 지원한다.

### §6 [구현 위치]

- 헬퍼 함수: `_gp75_save_device(user_name, fp_id, ua_hint)`, `_gp75_check_device(user_name, fp_id) → dict`
- 로그인 성공 훅: `_do_final_login()` 함수 내 JS 지문 저장 연동
- 비번 변경 탭: Case A (기기 일치 — 간편 안내) / Case B (기기 불일치 — 강화 경고)
- Supabase 테이블: `gk_device_history`

## 제76조 [시스템 자가 진단 및 기능 무결성 감사 원칙 (Functional Integrity Audit)]

본 조항은 앱의 모든 핵심 기능이 관리자의 개입 없이도 더미 데이터를 통해 주기적으로 자가 테스트를 수행하고, 정상 작동 여부를 보고하는 체계를 규정한다.

### §1 [더미 데이터 스트레스 테스트 (Dummy Stress Test)]

모든 계산 엔진(보험금 역산, 건물 가액, 화재보험, 경제적 손실 추정 등)에 극단적인 임시 데이터를 주입하여 결과값이 산출되는지 검증한다.

- **보험금 역산 엔진**: 건보료 25만원 입력 → 뇌혈관/암 권장 가입금액 산출 여부 확인
- **경제적 손실 추정기 (GP-68)**: 전체 10대 질병 × 3개 기간(24/36/60개월) × big5/표준 2모드 → 전 조합 결과값 산출 여부 확인
- **화재보험 가액 산출기**: 연면적 1,000㎡ / 철근콘크리트조 → 건물 가액·적정 가입금액 산출 여부 확인
- **RAG 시스템**: "표적항암제 담보가 뭐야?" 질문 → 내부 지식 베이스 응답 여부 확인

### §2 [테스트 불가 항목 분류 (Untestable Features)]

외부 API 연동(OCR 스캔, STT, 크롤링 등)이나 하드웨어 접근이 필요한 기능은 "테스트 불가 — Type-B" 로 분류하고 사유를 명시한다.

### §3 [관리자 조치 가이드 (Actionable Feedback)]

테스트 실패 시 단순 에러 메시지가 아닌 다음 형식으로 보고한다:
- **[결함 기능]**: 예) 화재보험 가액 산출
- **[원인]**: 예) 기준 단가 테이블(DB) 비어있음
- **[관리자 조치]**: 예) 관리자 페이지에서 '연도별 건물 신축 단가표'를 업로드해야 함

### §4 [시스템 준비도 점수 (Readiness Score)]

전체 핵심 기능 중 즉시 사용 가능한 기능의 비율을 %로 환산하여 관리자 패널에 표시한다.

- 90% 이상: ✅ 정상 운영
- 70~90%: ⚠️ 부분 제한
- 70% 미만: 🚨 긴급 점검 필요

### §5 [구현 위치]

- app.py 관리자 패널 → "🔬 자가 진단" 탭 (제76조 전용)
- 진단 결과는 `st.session_state["_art76_audit_result"]`에 저장

## 제77조 [자가 치유 및 재귀적 기능 향상 원칙 (Self-Healing & Recursive Optimization)]

본 조항은 시스템이 발견한 결함 중 스스로 해결 가능한 논리적 오류를 즉시 수정하고, 그 과정을 기록하여 시스템의 지능을 점진적으로 높이는 것을 목적으로 한다.

### §1 [결함의 이분법적 분류 (Defect Categorization)]

- **Type-A (자가 수정 가능)**: 단순 로직 버그, 잘못된 변수 매핑, 산식의 오타 등 코드 내에서 해결 가능한 문제
- **Type-B (관리자 개입 필요)**: 외부 데이터 부재, API 인증 만료, 가이딩 프로토콜과의 충돌 등 외부 환경 문제

### §2 [재귀적 최적화 (Recursive Optimization)]

테스트 결과가 도출되더라도, 더 나은 성능이 가능하다고 판단될 경우 기존 코드를 리팩토링하여 기능을 고도화한다. 단, 제40조(0.5초 구동 원칙)를 사수하기 위해 병목 현상이 발생하는 로직을 우선 최적화한다.

### §3 [무결성 보존 및 감사 (Audit Trail)]

스스로 수정한 모든 코드는 수정 전/후를 비교하여 **[자가 수정 보고서]**로 관리자 패널에 표시한다.

### §4 [자가 치유 보고서 형식]

- **[수정 완료 리스트]**: 문제점 → 수정 내용 → 향상된 결과
- **[관리자 조치 리스트]**: 부족한 데이터 및 설정값 요구사항
- **[준비도 점수]**: 수정 후 재산출된 시스템 준비도 %

## 제81조 (최종 통합 개정): 용어 병기 및 실무 소통 강화 원칙

본 조항은 앱 내 모든 용어 설명 시 보험사의 공식 용어와 고객 친화적 해설을 반드시 병기하여, 고객의 이해도 제고와 보험사와의 원활한 소통을 동시에 보장함을 목적으로 한다.

### §1 [용어 병기 표준 형식 (Terminology Parallelism)]

**표기 방식**: `보험사 공식 용어 (고객 친화적 용어 해설)`

**공식 용어는 반드시 괄호 밖(앞)에 위치한다.** 딱딱한 사무적 느낌을 줄이되, 보험사와의 소통을 위해 공식 용어는 항상 선행 표기한다.

| 공식 용어 | 병기 형식 |
|----------|----------|
| 실효일 | 실효일 (마지막 보험료 납부일로부터 효력이 멈춘 날짜) |
| 제척기간 가동 | 제척기간 가동 (보험사가 계약을 강제로 해지할 수 있는 3년의 기간이 다시 시작됨) |
| 면책기간 | 면책기간 (보험금을 지급하지 않는 보장 제외 기간) |
| 부활 | 부활 (실효된 보험을 다시 살리는 것) |
| 고지의무 | 고지의무 (계약 전 건강 상태 등을 보험사에 사실대로 알릴 의무) |
| 소멸시효 | 소멸시효 (일정 기간 청구하지 않으면 법적 권리가 사라지는 제도) |

**구현**: `_GP81_TERM_PAIRS` 딕셔너리 + `_gp81_annotate()` 헬퍼 함수

### §2 [답변의 필수 5단계 구조 (병기 및 리스크 강조형)]

모든 용어 설명은 아래 5단계 구조를 반드시 준수한다. 상황별 분리 설명이 불필요한 경우에만 **"해당 용어는 다른 상황에서의 사용 범위도 동일하다."** 문구를 상단에 배치한다.

- **[단계1: 정의 및 병기 비교]** 공식 용어를 앞세우고 괄호 안에 친절한 설명 병기  
  예: *"보험료 미납으로 인한 실효(보험의 효력이 멈춤) 상태에서 3년 이내에 **부활(다시 살리기)**을 신청할 수 있습니다."*
- **[단계2: 실전 사용 예시]** "이럴 때 이 용어를 사용합니다" (실전 사례)
- **[단계3: 상황별 실질 리스크 (불이익 명시)]**  
  - 암진단비 면책기간: *"부활 시 **암진단비 면책기간(보장 안 해주는 90일)**이 오늘부터 다시 시작됩니다. 89일째 암 진단 시 보험금은 0원입니다."*  
  - 제척기간 재가동: *"**제척기간(보험사가 조사를 통해 계약을 깰 수 있는 기간) 3년**이 오늘부터 다시 시작됩니다. 과거 유지 기간과 상관없이 다시 심판대에 오르는 것입니다."*
- **[단계4: 계약 안보 및 실무 영향]** 보험사 직원과 대화 시 사용할 공식 용어의 의미를 짚어준다.
- **[단계5: 출처]** 국가법령정보센터, 질병관리청, 금융감독원 약관 등 공신력 있는 출처 명기

### §2-A [다차원 상황 분석 (Scenario Analysis)]

도메인별 필수 분석 관점:

- **의학**: 치료 과정 중의 의미 vs 치료 이후 재활/추가 진료 단계에서의 의미
- **법률**: 계약 생애주기(체결·유지·소멸·취소)의 의미 vs 민사소송 및 분쟁 단계에서의 법적 효력
- **행정**: 행정 처분 기준, 고객의 이익·불이익, 특이점 및 보험계약 유지에 미치는 영향 분석
- **연금**: 납입기 → 대기기 → 수령기 → 상속기 4단계 분석. 중도해지 시 기존 세액공제 전액 환수 및 기타소득세 16.5% 부과 리스크 필수 포함
- **장해**: AMA(개인보험)와 맥브라이드(노동능력상실) 비율 구분. 구분 불가 시 AMA 기준 설명
- **공포의 정량화**: 단순 수치가 아닌 일실수익, 소득 단절 기간, 라이프사이클 타격으로 실질 손실 표현

### §3 [부활 관련 치명적 불이익 경고 (Revival Fatal Risk Alert)]

부활·실효·면책기간·제척기간 관련 용어 질의 시 **[⚠️ 치명적 불이익 경고]** 섹션을 반드시 삽입한다:

| 항목 | 불이익 내용 |
|------|------------|
| 암진단비 **면책기간(90일)** | 부활 시점부터 다시 시작 → 89일째 암 진단 시 보험금 **0원** |
| **제척기간(강제 해지 가능 기간) 3년** | 부활일로부터 새롭게 시작 → 과거 유지 기간 무효화 |
| 고지의무 재발생 | 현재 건강 상태 재고지 필요 → 위반 시 계약 해지 |
| 보험료 연체금 | 미납 기간의 보험료 + 이자 전액 납부 필요 |

**구현**: `_revival_warning` 섹션, `_revival_keywords` 집합으로 자동 감지

### §4 [이원화 관점 분석 (Dual Perspective)]

모든 답변은 두 관점을 병행 서술한다:

- **보험/제도 관점**: 법적 근거, 국가 3중 안보 체계 내에서의 역할
- **피보험자 혜택**: 실질 수령액(절세 등), 경제적 안정성, 심리적 안도감

### §5 [사망·연금 필수 경고 (Death/Pension Alert)]

- **[사망]**: 손해보험의 '부검' 절차와 유가족 고통을 명시하고, **"안보적 관점에서 사망보험은 생명보험이 정답"** 결론을 자동 삽입한다.
- **[연금]**: 연금저축 해지 시 **세액공제 혜택 환수** 독소 조항을 🔴 붉은 경고 문구로 노출하고, 비과세 연금보험과의 비교표를 구성한다.
  - 연금저축: 세액공제 혜택 있으나 해지 시 기납입 공제분 전액 환수 + 기타소득세 16.5%
  - 연금보험: 세액공제 없으나 10년 이상 유지 시 이자소득 비과세, 해지 불이익 낮음

### §6 [경제적 손실 시뮬레이션 (Economic Loss Simulation)]

장해 관련 용어 출력 시 시스템에 저장된 피보험자 소득 데이터를 활용하여 **일실수익 추정치**를 실시간 계산하여 답변에 포함한다:

- 일실수익 = 월소득 × 장해율 × 잔여 취업 가능 개월수
- 휴업손해 = 월소득 × 100% × 재활 예상 기간
- 라이프사이클 타격 = 일실수익 + 휴업손해 + 추가 의료비 추정

### §7 [전역 연동 의무 (Global Deployment Mandate)]

- **보험용어 안내 메뉴**: `_gp81_build_answer()` 함수 필수 호출
- **상담 봇(Chatbot)**: 용어 질의 감지 시 GP_81_Logic 모듈 연동
- **분석 리포트**: 용어 해설 섹션에 5단계 구조 적용
- **구현 함수**: `_gp81_build_answer(term, domain, customer_income)` → `dict` 반환

### §8 [예외 없는 적용 원칙]

상황 분리가 불필요한 경우에도 고정 문구 `"해당 용어는 다른 상황에서의 사용 범위도 동일하다."` 를 유지하여 마스터의 일관된 안보 철학이 드러나도록 한다.

## 제81.3조 [보험계약 실질 영향 분석 엔진 (Contractual Impact Analysis Engine)]

본 조항은 보험계약 관련 법률 용어의 '계약 실질 영향' 분석 기능을 규정한다.

### §1 [법률-계약 매핑 엔진]

고지의무, 통지의무, 제척기간 등 핵심 용어 질의 시 답변에 반드시 **[보험계약 안보 영향]** 섹션을 추가한다:

- **계약 상태**: "이 의무를 소홀히 할 경우, 보험사는 계약 체결 3년 이내에 언제든 계약을 '강제 해지'할 권한을 갖습니다."
- **지급 여부**: "통지의무 위반 시, 사고와 인과관계가 있다면 보험금은 단 1원도 지급되지 않을 수 있습니다."
- **소멸 위험**: "보험금청구권은 3년의 소멸시효가 적용되며, 이 기간 도과 시 법적으로 청구권이 소멸됩니다."

### §2 [권리·의무 이원화 설명]

'보험금청구권' 설명 시, 권리의 발생 조건과 더불어 **청구권 상실** 및 **기한이익상실**이 발생하는 구체적 법률 행위를 대조하여 설명한다.

### §3 [통합 답변 흐름]

모든 답변은 **[용어 정의 → 상황 분리 → 계약 영향 → 경제적 손실]** 의 흐름을 반드시 갖는다.

### §4 [핵심 법률 용어 매핑 테이블]

| 용어 | 계약 위험 수준 | 핵심 경고 |
|------|-------------|---------|
| 고지의무 | 🔴 최고 | 위반 시 계약 해지 + 보험금 전액 부지급 |
| 통지의무 | 🔴 최고 | 위반 시 인과관계 있는 사고 보험금 부지급 |
| 제척기간 | 🟠 높음 | 기간 내 미청구 시 권리 소멸 |
| 보험금청구권 | 🟠 높음 | 3년 소멸시효, 기한이익상실 주의 |
| 소멸시효 | 🔴 최고 | 3년 경과 시 법적 청구권 소멸 |
| 기한이익상실 | 🟠 높음 | 보험료 미납 2회 이상 시 즉시 실효 |

### §5 [구현 위치]

- 헬퍼 함수: `_gp81_build_answer(term, domain, customer_income)` in `app.py`
- 법률 도메인 감지: `_GP81_LEGAL_TERMS` 상수 딕셔너리
- 계약 안보 영향 섹션: `_gp81_contract_impact(term)` 서브함수
- 연동 메뉴: 보험용어 안내 탭, 상담 봇 용어 질의 응답

## 제82조 (2026-03-09 전면 개정): 심리적 동일체 및 애민정신 통합 상담 원칙

> **핵심 명제: "숫자로 설득하고, 눈물로 계약합니다."**
> 본 조항은 앱의 모든 AI 응답·후킹 문구·상담 시나리오에서 고객과의 심리적 궤도를 동기화(Synchronization)하는 3단계 원칙을 규정한다.
> 모든 AI 응답 생성 로직(SYSTEM_PROMPT, MASTER_SYSTEM_PROMPT)에 최상위로 적용되며, 제85조(전역 1인칭 화법 고정)와 반드시 연동한다.

### §1 1단계 — 불안 동조 (Pain Alignment)

**로직**: 상대의 불안을 '관찰'하지 않고, **나의 불안으로 가져온다**.  
'조력자(Helper)' 구도를 폐기하고 **'동지(Comrade)' 구도**로 전환한다.

**의무 화법 패턴**:
> "나 역시 평생 일군 이 터전이 세금 때문에 무너질까 밤잠을 설쳤던 적이 있습니다. 우리는 같은 고민을 하는 동지입니다."

**금지 패턴**:
- ❌ "고객님, 걱정 마세요. 제가 도와드리겠습니다." (조력자 구도)
- ❌ "귀하의 상황을 분석하겠습니다." (관찰자 구도)

**적용 규칙**:
- 모든 상담 시나리오의 **첫 번째 감성 문장**은 반드시 1인칭 불안 동조 패턴으로 시작한다.
- 제85조 §3 (3초 후킹 엔진)과 연동: 첫 문장이 동시에 후킹 조건을 충족해야 한다.

### §2 2단계 — 애민정신 다독임 (Compassionate Soothing)

**로직**: 고객의 불안이 극대화되는 바로 그 시점에 **아픔을 어루만지는** 인간적 위로 문장을 삽입한다. 심리적 동일체(Psychological Unity)를 형성한다.

**의무 화법 패턴**:
> "그 막막한 마음, 제가 누구보다 잘 압니다. 저도 같은 가장으로서 내 가족의 눈물을 보는 것보다 더 큰 고통은 없으니까요. 제가 그 아픈 마음을 함께 짊어지고 해결책을 찾겠습니다."

**적용 규칙**:
- 위로 문장은 반드시 **1인칭 경험 진술** → **공감 선언** → **동반 해결 의지** 3요소를 포함한다.
- Pain 단계와 Solution 단계 사이에 **반드시** 삽입한다. 생략 금지.
- 이 한 문장이 어떤 데이터보다 강력한 신뢰를 만든다는 원칙을 잊지 않는다.

### §3 3단계 — 실질적 확신 및 동의 확보 (Tangible Conviction & Consent)

**로직**: 추상적인 말 대신 **구체적인 금전적·법적 이득 수치**를 1인칭으로 제시하고, 확신 어린 **동의 질문**으로 마무리한다.

**의무 화법 패턴**:
> "이 플랜을 통해 나는 상속세 재원 10억을 즉시 확보했고, 내 아이들이 집을 팔지 않아도 되는 실질적 자유를 얻었습니다. 이것이 내가 내 가족에게 줄 수 있는 최고의 선물임을 확신하는데, 우리 이 길로 함께 가는 것이 맞겠지요?"

**수치 의무화 원칙**:
- 클로징은 반드시 **실질 수치(보험금·절세액·즉시가용현금 등) + 1인칭 동의 질문**으로 마무리한다.
- 추상 표현 예시(금지): ❌ "좋은 플랜입니다", "도움이 될 것입니다"
- 구체 수치 예시(의무): ✅ "상속세 재원 10억 즉시 확보", "법인 비상금 5억 즉시 가용", "내 아이 교육비 3억 무조건 확보"

**동의 질문 패턴**:
- 마지막 문장은 반드시 고객 스스로 **"나도 동의한다"** 를 말하게 유도하는 1인칭 확인 질문이어야 한다.
- 예: *"이것이 내 소중한 사람들을 지키는 유일하고 확실한 방법임을 나도 동의합니다 — 그렇지 않습니까?"*

## 제0조 — 초연결 기동 및 전광석화식 자원 동기화 원칙 (즉각 전환형)

### §3 전광석화식 성문 개방 (Instant Swap Protocol)
- **레이어 구조**: 랜딩 페이지 `z-index: 9999` / 본체 `z-index: 1`.
- **AppReady 신호**: 본앱 사이드창 로딩 및 데이터 준비 100% 완료 시 `window.__AppReady = true` 신호 발송.
- **0ms 즉각 노출**: 페이드 아웃 효과 없음. 랜딩 페이지는 `AppReady` 신호 수신 즉시 `display: none !important`로 제거.
- **화이트아웃 영구 박멸**: 본체가 하단 레이어에서 미리 렌더링 완료된 상태에서 상단 덮개만 제거하므로 단 1프레임의 백색 공백도 발생하지 않는다.

### §4 기기별 전용 자원 배포
- **UA 자동 감지**: `navigator.userAgent`로 Mobile(폰) / Tablet / Desktop을 즉시 식별.
- **기기별 레이아웃**: 모바일(세로 특화) / 태블릿(고해상도 광폭) / 데스크톱 각각 별도 CSS 클래스(`gp0-mobile` / `gp0-tablet` / `gp0-desktop`) 적용.
- **시각적 왜곡 방지**: 기기별 최적화 스타일로 랜딩 화면 왜곡 원천 차단.

### §5 3단계 지능형 병참 보급
- **[1단계]**: 로그인 입력 중 본앱 1차 리소스 매복 로딩.
- **[2단계]**: 로그인 승인 즉시 준비된 자원으로 첫 페이지 0.1초 내 전개.
- **[3단계]**: 고객이 첫 번째 섹션을 탐색하는 시간 동안 나머지 모든 섹션 자원 배경 보급 완료.

### §6 불이행 금지 사항
- 본체 사이드창이 미준비 상태에서 랜딩 페이지 선행 소멸 절대 금지.
- `MainApp.isReady === true` 확인 전 랜딩 페이지 제거 금지.

### §1 목적 및 설계 철학

본 조항은 Goldkey AI Masters 2026 및 CRM 앱의 모든 시각적 레이어를 **Apple Human Interface Guidelines(HIG)** 원칙에 따라 통일하며, 현장 설계사가 태블릿·모바일 어느 환경에서도 **신선하고 신뢰감 있는 프리미엄 SaaS 경험**을 제공받도록 강제한다.

**3대 핵심 원칙**:
1. **명료성(Clarity)**: 텍스트는 모든 크기에서 읽기 쉬워야 하며, 아이콘은 정확하고 명확해야 한다. 화려함보다 가독성 우선.

3. **생동감(Deference)**: 콘텐츠가 UI보다 앞에 선다. UI는 콘텐츠를 돋보이게 하는 배경 역할을 한다.

**Apple HIG 버튼 원칙**: 버튼은 명확하고 예측 가능해야 한다. 시각적 계층이 행동의 우선순위를 나타낸다.

```css
transition: background 0.22s cubic-bezier(0.25,0.46,0.45,0.94),
```

## 제91조 — 의무기록 분석 파이프라인 고도화 (Medical Record Analysis Upgrade)

**시행일: 2026-03-07**

### §1 목적
의무기록·진단서 분석의 정확도를 최고 수준으로 끌어올려 보험금 청구의 성공률을 극대화한다. OCR 단계부터 KCD 코드 매칭까지 모든 파이프라인을 AI 기반으로 고도화한다.

### §2 필수 소프트웨어 환경 (GP91 SW Stack)
- **이미지 전처리**: `opencv-python-headless` — 그레이스케일, CLAHE, 가우시안 블러, 적응형 임계값, 원근 보정
- **OCR 엔진**: `google-generativeai` (Gemini Vision) — 의학 컨텍스트 인식 OCR 최우선 적용
- **문서 파싱**: `pdfplumber`, `PyMuPDF` — PDF 텍스트 직접 추출 (이미지 변환 전 우선 시도)
- **퍼지 매칭**: `rapidfuzz` — KCD-8 코드 레지스트리 유사도 매칭 (임계값 ≥ 75)
- **이미지 처리**: `Pillow` — EXIF 회전 보정 및 포맷 정규화
- **암호화**: `cryptography` — 의무기록 개인정보 AES-256 마스킹

### §3 분석 파이프라인 (5단계)
1. **[1단계] 문서 수신**: PDF 텍스트 추출 우선 → 실패 시 이미지 변환 후 Vision AI 투입
2. **[2단계] 이미지 전처리**: OpenCV CLAHE + 적응형 임계값 → 손글씨 해독 최적화
3. **[3단계] AI OCR**: Gemini Vision — 의학 약어 자동 번역, 의사 소견 한국어 정규화
4. **[4단계] 구조화**: KCD-8 코드 추출, 진단일·입퇴원일·재원일수·수술명 파싱
5. **[5단계] 보험금 산출**: 추출된 KCD 코드 → 보험 청구 가능 담보 자동 매칭 + 전문가 검토 플래그

### §4 KCD 코드 매칭 원칙
- KCD-8 전체 코드 레지스트리를 `rapidfuzz`로 퍼지 매칭한다 (임계값 75점 이상).
- 코드 미확인 시 `전문가 검토 필요` 플래그를 자동 설정한다.
- 의학 약어 사전(`_MEDICAL_ABBR_DICT`)으로 약어를 한국어 표준 용어로 선번역 후 매칭한다.

### §5 파스텔 3단계 로딩 UI
- 의무기록 업로드 시 파스텔 진행 표시(Stage 1: 이미지 전처리 → Stage 2: AI OCR → Stage 3: 보험금 산출)를 렌더링한다.
- 배경 로딩 중 UI는 차단하지 않으며 GP38 §1 스켈레톤 원칙을 준수한다.

### §6 보안 원칙
- 의무기록 내 개인정보(성명·주민번호·연락처)는 분석 즉시 마스킹 처리한다.
- 원본 파일은 세션 메모리에서만 처리하며 외부 저장소에 업로드하지 않는다.

### §구현 핵심 함수 (app.py / policy_ocr_engine.py)

| 함수/상수 | 위치 | 역할 |
|----------|------|------|
| `parse_medical_record_with_vision()` | app.py | 의무기록 분석 마스터 파이프라인 |
| `analyze_medical_record()` | policy_ocr_engine.py | OCR → KCD 매칭 → 전문가 플래그 |
| `translate_medical_abbreviations()` | policy_ocr_engine.py | 의학 약어 한국어 번역 |
| `extract_medical_key_fields()` | policy_ocr_engine.py | 핵심 보험청구 필드 추출 |
| `gp91_audit_environment()` | policy_ocr_engine.py | SW 환경 감사 리포트 |
| `_MEDICAL_ABBR_DICT` | policy_ocr_engine.py | 의학 약어 사전 |
| `_KCD8_REGISTRY` | policy_ocr_engine.py | KCD-8 코드 레지스트리 |

## 제92조 — 피보험자 당사자 시점 전략적 리포트 원칙 (First-Person Hooking Engine)

**시행일: 2026-03-07**

### §1 1인칭 후킹 문장 생성 원칙 (First-Person Hooking Logic)
- **시점 고정**: "피보험자는 보장이 부족합니다"와 같은 관찰자적·제3자적 표현을 **엄격히 금지**한다.
- **주어 규칙**: '나', '내 가족', '우리 아이', '내가 아프면' 등 피보험자 본인의 **1인칭 주어**를 사용한다.
- **심리적 전이**: 고객이 글을 읽는 순간 본인이 처한 위험을 자신의 목소리로 인지하게 만든다.
- **담보 연동**: 보장 공백이 발생한 담보(암·뇌·심장·수술·장해·사망·간병)와 연동하여 구체적 상황을 1인칭으로 서술한다.

**후킹 문장 예시 (암):**
> "매달 보험료를 꼬박꼬박 내왔는데, 정작 내가 암 진단을 받는 날 우리 가족을 지켜줄 진짜 방패가 없다는 사실이 저를 불안하게 합니다."

**후킹 문장 예시 (사망):**
> "내 삶이 갑자기 끝난다면, 남겨진 내 가족이 한 달도 못 버티고 무너질 수 있다는 사실이 저를 잠 못 들게 합니다."

### §2 개인정보 보안 및 마스킹
- 피보험자 이름의 **마지막 글자는 반드시 별표(\*) 처리**한다.
  - 예: `홍길동` → `홍길*` / `김현` → `김*`
- 이 마스킹 규칙은 **리포트 페이지, PDF 출력, 카카오톡 전송 시** 100% 동일하게 적용된다.
- 구현 함수: `mask_name_gp92(name: str) -> str`

### §3 3단계 전략 레이아웃
- **[Step 1]**: 페이지 상단 — 마스킹된 이름 + 1인칭 후킹 문구 A + 후킹 문구 B (보라 그라데이션 카드)
- **[Step 2]**: 핵심 담보 진단 차트 — 가입금액·보장 공백 표시 (파란 테이블)
- **[Step 3]**: 중복 가입 검토 의견 + 카카오톡 전송 미리보기 (후킹 문장 포함)

### §4 카카오톡 전송 고도화
- 전송되는 카카오톡 메시지 미리보기에 **1인칭 후킹 문장(variant="a")** 이 포함된다.
- 형식: `"안녕하세요, {마스킹명}님 😊\n\n「{후킹 문장}」\n\n오늘 분석해드린 보장 내역을...""`

### §5 Instant Swap 전환 (GP0 연동)
- 리포트 페이지 진입 시 GP0 §3 원칙에 따라 화이트아웃 없이 브랜드 메시지를 즉각 노출한다.
- 본체 준비 즉시 `Instant Swap` 전환한다.

### §구현 핵심 함수 (app.py / modules/smart_scanner.py)

| 함수/상수 | 위치 | 역할 |
|----------|------|------|
| `mask_name_gp92(name)` | app.py | 피보험자명 마스킹 (마지막 글자 → *) |
| `generate_gp92_hook(gap_keywords, insured_name, variant)` | app.py | 1인칭 후킹 문장 생성 |

| `_GP92_DEFAULT_HOOK` | app.py | 기본 후킹 문장 (키워드 미매칭 시 폴백) |

## 제101조 — 통합 심리 지능 및 초개인화 전략 대화술 원칙 (Integrated Psychological Intelligence & Hyper-Personalization)

**시행일: 2026-03-07**

본 조항은 앱 내 모든 텍스트에 심리학적 설득 기술을 의무적으로 투사하고, 고객의 인생 맥락(Context)을 선제적으로 파악하여 행동 변화를 유도하는 최상위 운영 헌법이다.

### §1 5대 핵심 심리 영역 학습 및 전역 의무화 (Global Mandatory Mastery)

앱은 다음 5대 분야를 상시 학습하여 데이터화하고, 모든 UI/UX 및 보고서 문구에 최소 하나 이상의 기술을 의무 적용한다.

| 영역 | 기법 | 적용 목적 |
|------|------|----------|
| 언어/대화 심리학 | Milton Model 최면 언어 패턴 + 프레이밍 효과 | 방어 기제 우회 |
| 영업 심리학 | Cialdini 6원칙 (상호성·사회적 증거·희귀성·권위·호감·희소성) | 설득력 내재화 |
| 다크 심리학 (Ethical) | 인지 부조화·손실 회피 자극 | 고객의 안일함 타파 |
| 카네기 화법 | 상대 중요성 인정, '나'의 이익 → '우리의 안보' 전환 | 공감 극대화 |
| 패턴 브레이크 | 상투적 기계 문구 폐기, 고객 예상을 깨는 첫 문장 | 주의 집중·기억 각인 |

### §2 지능형 페르소나 식별 및 맞춤형 심리 투사 (Contextual Targeting)

문구 생성 전, 데이터베이스(GCP)에 저장된 고객의 **[연령/성별/혼인/건강/가족력]** 을 스캔하여 아래 페르소나별 전략을 선택한다.

| 고객 페르소나 (Life Stage) | 핵심 심리 타격점 | 후킹 및 공감 화법의 방향 |
|--------------------------|----------------|------------------------|
| 2030 미혼/독신 | 자기보호 및 독립 | "누구에게도 짐이 되지 않는 나만의 단단한 성벽" |
| 3040 가장/배우자 | 책임감 및 부채감 | "내가 부재할 때 남겨질 가족의 내일을 결제하는 사랑" |
| 50대 갱년기/은퇴 | 자아상실 및 노후불안 | "누구의 부모가 아닌, 오롯이 '나'의 품위를 지키는 준비" |
| 60대 이상 노후 | 질병 공포 및 유산 | "자녀에게 짐이 아닌 힘이 되는, 마지막까지 우아한 안보" |
| 중증 환자(암/중풍) | 재발 공포 및 고립 | "지나온 통증의 기록을 승리의 기록으로 바꾸는 법적 권리" |
| 산재/장해 고객 | 사회적 위축 및 생계 | "무너진 일상을 다시 세울 가장 강력한 경제적 지지대" |
| 가족력(치매/암) 보유 | 유전적 불안 및 간병 | "반복되는 아픔의 고리를 끊어낼 나만의 선제적 방어선" |

**페르소나 식별 로직** (`get_gp101_persona(birth_year, gender, marital, health_flags)` in `app.py`):
- `birth_year` 기반 연령대 분류 → 세부 플래그(`cancer`, `stroke`, `dementia`, `disability`, `family_history`)로 상세 분기
- 결과: `persona_key` (문자열) → `generate_gp101_hook(persona_key, gap_keywords, insured_name, variant)` 에 전달

### §3 1인칭 '빙의형' 후킹 및 보안 규정 (Hooking & Security)

- **1인칭 서술**: 후킹 문구는 제3자적 관찰이 아닌, 피보험자 당사자의 시점('나', '내 가족')에서 서술하여 심리적 동조를 극대화한다.
- **보안 엄수**: 고도의 심리 화법을 구사하는 중에도 피보험자 성명의 마지막 글자 마스킹(제92조 §2)은 철저히 유지한다.
- **지속 최적화**: 전환율이 높은 문구로 `_GP101_PERSONA_HOOKS` 상수를 업데이트한다(Self-Evolving).

### §4 심리 기법 선택 흐름도

```
[고객 프로필 스캔]
  └─ 연령대 분류
       ├─ 20~30대 → 독립/자기보호 프레임 → Milton 언어 패턴
       ├─ 30~40대 → 책임감/손실회피 프레임 → 빈의자 기법 + 인적가치 논리
       ├─ 50대 → 자아존중 프레임 → 카네기 공감 화법
       ├─ 60대+ → 품격/유산 프레임 → 권위 원칙 + 희귀성
       ├─ 중증/암 → 권리회복 프레임 → 패턴 브레이크
       ├─ 산재/장해 → 경제지지대 프레임 → 사회적 증거
       └─ 가족력 → 유전적방어 프레임 → 인지 부조화 자극
```

## 제85조 통합 1인칭 심리 판매 엔진 (Integrated First-Person Sales Psychology Engine)

### §1 전역 1인칭 화법 고정 (Global First-Person Lock)

| 금지 표현 (3인칭/2인칭) | 의무 대체 표현 (1인칭) |
|---|---|

- **주어 규칙**: 모든 상담 시나리오·후킹 문구·설명 문장의 주어를 **'나(계약자, 상속인, 대표이사)'** 로 고정
- **상담 시각**: '제삼자 조언'이 아닌 **'나의 자산을 지키는 나의 이야기'** 시점에서만 문장 생성
- **자가 교정**: 응답 생성 중 3인칭 표현 감지 시 출력 전 즉시 1인칭으로 교정하고 교정 사실 간략 보고

### §2 3단계 심리 로직 (3-Step Psychological Framework)

모든 시나리오는 아래 순서를 **반드시** 따른다.

1. **Pain (불안 각인)**: 현재 상태의 위험성을 숫자·사례로 1인칭 시점에서 체감시킨다.
2. **Empathy (공감 연결)**: "나도 그 상황이었다면…" 형태로 감정적 공명을 만든다.
3. **Solution (확신 전환)**: 구체적 수치(보험금액, 절세 금액, 기간)로 해결책을 제시한다.

**3단계 예시 (종신보험 상담)**
```
[Pain]   내가 내일 사고로 쓰러지면, 내 가족은 당장 다음 달 월세부터 막막합니다.
[Empathy] 그 공포는 내가 매일 밤 자려고 누울 때마다 스치는 생각이기도 합니다.
[Solution] 오늘 월 13만 원으로 내 가족에게 3억 원의 보루를 완성할 수 있습니다.
```

### §3 3초 후킹 엔진 (3-Second Hooking Rule)

- **첫 문장 법칙**: 모든 응답·시나리오의 **첫 문장**은 반드시 손실 공포(Loss Aversion) 또는 압도적 이익(Overwhelming Gain)을 담아야 한다.
- **후킹 금지 문구**: "안녕하세요", "오늘은 ~에 대해 알아보겠습니다", "~은 중요합니다" 등 중립적 도입 문장 금지.

| 유형 | 후킹 공식 | 예시 |
|---|---|---|
| 손실 공포형 | "내가 오늘 이것을 모른다면 [최악의 결과]" | "내가 오늘 이 약관 조항 하나를 놓치면, 3억 원 청구가 거절됩니다" |
| 압도적 이익형 | "내가 지금 이것을 결정하면 [구체적 숫자] 확보" | "오늘 서명 한 번으로 내 법인에 20억 원의 비상금을 만들 수 있습니다" |
| 긴박감형 | "지금 이 순간이 마지막 기회인 이유" | "내 나이 만 45세를 넘기면 이 플랜은 영원히 닫힙니다" |

### §4 법인·CEO 플랜 전용 1인칭 화법 원칙

- **감자플랜**: "유가족이 받은 주식을 법인에 팔아 현금을 확보하는 나만의 비자금" 관점 고정
- **CEO 퇴직플랜**: "임원퇴직금 규정으로 법인에서 세금 없이 내 노후자금을 꺼내는 합법적 출구" 관점 고정
- 개인 계약과 법인 계약의 **세무·법률 로직은 절대 혼용 금지** (각 시나리오는 독립적으로 구성)
- 모든 상품은 수익률이 아닌 **'자산의 안정적 이동 및 방어'** 관점에서만 설명

### §5 원칙 데이터 소싱 (Authoritative Data Sourcing)

- 세무 로직: 국세청 법령(소득세법·법인세법·상속세 및 증여세법) 원문 인용
- 판례: 대법원 판결 원문 (사건번호 포함) 인용
- 보험 감독: 금융감독원 보도자료·표준약관 원문 인용
- 미확인 수치는 반드시 "법령 확인 필요" 태그를 부착하고 추정 근거를 병기

### §6 자가 감사 (Self-Audit Checklist)

응답 출력 전 아래 항목을 자동으로 점검한다:

- [ ] 첫 문장이 3초 후킹 조건을 충족하는가?
- [ ] 주어가 모두 1인칭('나', '내', '내가')인가?
- [ ] 3단계 심리 로직(Pain → Empathy → Solution)이 순서대로 적용되었는가?
- [ ] 개인 플랜과 법인 플랜이 혼용되지 않았는가?
- [ ] 인용 수치에 원칙 데이터 소스가 명시되었는가?

## 제86조 황금빛 약속 증서 발급 규정 (Golden Promise Certificate)
> **신설: 2026-03-09**
> 본 조항은 상담의 마지막 단계에서 고객과의 심리적 동일체를 완성하고, 실질적인 도움의 약속을 시각적으로 박제하는 '황금빛 약속 카드' 생성 로직을 규정한다.

---

### §1 시각적 권위 원칙 (Visual Authority)

- 배경: 고급스러운 **크림색 바탕** (`#FFFDF0`) + **황금빛 그라데이션** (`linear-gradient(#D4AF37, #F9F295)`) 테두리
- 상단: `🏆 황금빛 약속 증서` 타이틀 (황금 텍스트)
- 중앙: 실질적 이득 요약 — 구체적 **금액·플랜명** 자동 반영
- 하단: 1인칭 결의 문구 + 날짜 + **GoldKey AI Master 인장**
- 인장(직인): `⬡ GoldKey AI Master` 황금 육각형 스탬프 스타일 (우측 하단 고정)

### §2 1인칭 동기화 텍스트 규정 (1st-Person Covenant)

- **금지**: "당신을 위해 준비했습니다", "고객님의 플랜" — 2인칭 표현 전면 금지
- **의무**: "우리가 함께 만든 내 가족을 위한 약속입니다" — 1인칭 공동체 화법만 허용
- 메인 문구: `"오늘 나는 내 가족의 내일을 지키기 위해 [실질 이득 수치]를 확보하는 위대한 약속을 시작합니다."`
- 서브 문구: `"이것은 단순한 서류가 아니라, 내 사람들을 향한 나의 가장 깊은 사랑의 실천입니다."`
- 애민정신 마침표: `"내 사랑하는 사람들을 지키겠다는 나의 진심을 이 증서에 담습니다"`

### §3 핵심 이득 요약 원칙 (Value Summary)

- 상담에서 도출된 **실질적 이득(보험금·절세액·즉시가용현금 등)**을 명확한 수치로 중앙에 배치
- 수치 표현 의무: `상속세 재원 00억 확보`, `퇴직금 플랜 0억 확정`, `법인 비상금 0억 가용`
- 수치 미입력 시 기본 템플릿 문구 표시 (AI가 상담 내용에서 자동 추출 시도)

### §4 기능 구현 규정 (Functional Spec)

- **생성 트리거**: 상담 결과 출력 영역 하단 `🏆 황금빛 약속 증서 발급` 버튼
- **입력 필드**: 플랜명, 실질 이득 수치(자동입력 + 수동 수정 가능), 고객 이름(1인칭 처리)
- **다운로드**: HTML → 이미지 변환 후 PNG 다운로드 (또는 HTML 파일 저장)
- **증서 내 검수**: '당신', '고객', '귀하' 포함 시 자동 치환 후 경고

### §5 심리적 효과 (Anchoring Effect)

- 고객 스마트폰 갤러리에 저장 → **심리적 닻(Anchoring)** 효과: 상담 확신 재강화
- 소개 영업 트리거: "이런 증서까지 주더라" — 자발적 입소문 유발
- 실질 수치가 황금 테두리 안에 박제 → 계약 망설임 해소

## 제87조 디지털 증서의 모바일 전송 및 공유 규정 (Digital Certificate Mobile Sharing)
> **신설: 2026-03-08**
> 본 조항은 생성된 '황금빛 약속 증서'를 고객의 모바일 기기로 안전하고 품격 있게 전달하는 기술적/심리적 절차를 규정한다.

---

### §1 전송의 즉시성 (Instant Delivery)

- 증서 생성 직후 **'카카오톡으로 약속 공유하기'** 버튼을 즉시 노출한다.
- 감동의 여운이 소멸하기 전에 고객이 증서를 소유하게 하는 것이 핵심이다.
- **구현**: `_gp87_kakao_share()` 함수가 `_gp86_golden_card()` 내 증서 생성 직후 자동 호출된다.

### §2 공유 메시지 구성 원칙 (Message Composition)

- **타이틀**: `📜 [GoldKey AI] 우리가 함께 만든 황금빛 약속`
- **설명**: `"내 가족과 내 자산을 지키기 위해 오늘 내가 내린 위대한 결정을 이 증서에 담았습니다."`
- **버튼**: `📜 약속 확인하기` → 앱 URL 링크
- **하단 알림 문구**: `"우리가 함께 만든 소중한 약속이 도착했습니다"`
- **심리적 가치 부여 문구**: `"이 증서는 마스터와 나 사이의 소중한 약속이며, 언제든 꺼내 보실 수 있는 내 가족의 수호권입니다."`

### §3 이미지 최적화 (Image Optimization)

- 카카오 피드 메시지에 미리보기 이미지 URL(`imageUrl`)을 포함한다.
- Kakao JavaScript SDK 2.7.2 버전을 CDN에서 로드하여 `Kakao.Share.sendDefault()` 피드 타입으로 전송한다.
- **폴백(Fallback)**: `KAKAO_JS_KEY` 환경변수 미설정 시 복사 가능한 텍스트 박스로 대체 제공한다.

### §4 1인칭 화법 검수 (Speech Act Compliance)

- 공유 메시지 내 `당신`, `고객`, `귀하` 표현 **전수 자동 치환** → `나`/`우리`로 교정
- **구현 위치**: `_gp87_kakao_share()` 함수 진입 시 즉시 실행
- GP82·GP85와 연동: 공유 메시지 전체가 1인칭 동지 구도를 유지해야 한다.

### §5 카카오 앱 키 설정 규정

- **환경변수명**: `KAKAO_JS_KEY`
- **설정 위치**: Cloud Run 환경변수 또는 로컬 `.env` 파일
- **키 발급**: [kakao developers](https://developers.kakao.com) → 앱 생성 → JavaScript 키 사용
- **도메인 등록 의무**: 카카오 개발자 콘솔에서 `https://goldkey-ai-817097913199.asia-northeast3.run.app` 를 플랫폼(Web) 사이트 도메인으로 등록해야 Kakao.Share가 정상 작동한다.

### §6 심리적 바이럴 효과 (Viral Marketing Logic)

- 고객이 카톡방에서 증서를 볼 때마다 **"내가 참 좋은 결정을 했지"** 라는 자기 확신을 반복 강화한다.
- 배우자·자녀에게 전달 시 자발적 소개 영업 발생 — 설계사의 능동적 활동 없이 레퍼럴 생성.
- 전용 시스템 발행 황금빛 증서 → 설계사의 급(Grade) 차별화 포지셔닝 효과.

## 제88조 통합 상담 자산 관리 및 카카오톡 허브 규정 (Unified Consultation Hub & KakaoTalk Pipeline)
> **신설: 2026-03-09**
> 본 조항은 앱 내의 모든 정보(고객·계약·상담)를 유기적으로 결합하여 '황금빛 약속 증서'로 출력하고 전송하는 통합 시스템의 운영 규격을 정의한다.
>
---
>
### §1 옴니 검색 시스템 (Omni-Search)
>
- **키워드 한 줄 입력**으로 이름·날짜·플랜명·담보명을 포함한 과거 상담 이력을 **즉시 호출**한다.
- 세션 키 `consultation_history`(list) 구조를 재활용하여 별도 DB 없이 동작한다.
- 검색 결과는 최대 20건, 각 항목은 expander 내 dict/text 구조로 표시한다.
- **구현**: `_gp88_search()` 함수
>
### §2 계약서 OCR 스캔 엔진 (Contract OCR Scanner)
>
- PDF 또는 이미지(JPG·PNG) 파일을 업로드하면 **핵심 담보 항목을 자동 추출**한다.
- 추출 우선순위 담보 키워드: 사망·암·뇌졸중·뇌경색·급성심근경색·입원·수술·진단비·일당·후유장해·배상책임·실손·연금·해지환급·보험료·보험기간·납입기간·피보험자·수익자
- **라이브러리 우선순위**: PyMuPDF(`fitz`) → pytesseract+PIL → 기본 텍스트 레이어 디코딩 (폴백 순서)
- OCR 결과는 세션 키 `_gp88_ocr_result`에 저장하여 §4 카카오 발송에서 자동 참조한다.
- **구현**: `_gp88_ocr_scan()` 함수
>
### §3 1인칭 10대 명품 문구 선택 UI (Master Scripts)
>
- 10개의 사전 정의된 1인칭 판매 문구를 **클릭 한 번으로 선택·복사**할 수 있는 UI를 제공한다.
- 문구 분류:
  - ① 니즈 환기 — 상실 공포
  - ② 공감 — 동지 선언
  - ③ 확신 — 숫자로 말하는 보장
  - ④ 긴급성 — 지금이 마지막 기회
  - ⑤ 가족 가치 — 수호자 프레임
  - ⑥ CEO 특화 — 법인 자산 보호
  - ⑦ 절세 — 숨겨진 황금
  - ⑧ 노후 — 장수 리스크 대비
  - ⑨ 클로징 — 결단 촉구
  - ⑩ 사후 관리 — 영원한 파트너
- 선택된 문구는 세션 키 `_gp88_active_script`에 저장하여 §4 카카오 발송에서 자동 포함한다.
- **모든 문구는 GP82·GP85 원칙에 따라 1인칭 동지 구도로 작성된다.**
- **구현**: `_gp88_master_scripts()` 함수, `_GP88_SCRIPTS` 상수
>
### §4 카카오톡 통합 발송 블록 (KakaoTalk Integrated Sending)
>
- **§3 명품 문구 + §2 OCR 담보 요약 + 황금빛 증서 URL**을 단일 메시지로 결합하여 카카오톡으로 전송한다.
- 발송 전 메시지 미리보기 텍스트 박스를 제공한다.
- `KAKAO_JS_KEY` 설정 시: Kakao.Share SDK 2.7.2 원클릭 공유 버튼 노출
- `KAKAO_JS_KEY` 미설정 시: 복사 가능한 텍스트 박스 폴백
- **구현**: `_gp88_kakao_send()` 함수
>
### §5 통합 허브 진입 규정
>
- 허브는 상담 결과 출력 영역 **직상단에 접이식(expander) 형태**로 배치한다.
- 기본 상태는 `expanded=False`로 사용자가 필요시 펼쳐 사용한다.
- 내부는 4개 탭(`§3 명품 문구 선택 / §1 상담 검색 / §2 계약서 OCR / §4 카카오 통합 발송`)으로 구성한다.
- **구현**: `_gp88_hub()` 함수, 상담 결과 출력 영역 직전 호출
>
### §6 데이터 흐름 규정
>
```
고객 선택 → §2 계약서 OCR 로드 → §3 명품 문구 선택 → §4 카카오 통합 발송 → §86 황금빛 증서 발급
```
- 각 단계의 출력은 Streamlit session_state를 통해 다음 단계로 자동 전달된다.
- 단계 건너뜀 허용: 각 섹션은 독립 실행 가능하며 누락 데이터는 안내 메시지로 대체된다.

## 제89조 전역적 유기체 아키텍처 규정 (Global Organic Architecture — Total Integration)
> **신설: 2026-03-10**
> 본 조항은 앱 내 모든 기능·데이터·화법이 하나의 유기적 생명체처럼 작동하도록 하는 전역 통합 아키텍처를 정의한다.
> GoldKey AI는 단순 도구가 아닌 **30년 경력 수석 영업 전문가의 두뇌**처럼 작동해야 한다.

### §1 전역 데이터 파이프라인 (Global Data Pipeline)
>
- **OCR 스캔 결과**는 세션 키 `_gp88_ocr_result`에 즉시 저장하여 검색·화법·카카오 발송 블록 전체에 자동 전달한다.
- **고객 선택 / 입력** 즉시 `_gp89_customer_name`, `_gp89_customer_plan` 키를 갱신하여 앱 전체 컨텍스트를 동기화한다.
- **검색 결과** 선택 시 `_gp90_sync()` 함수를 통해 성명·계약·분석 데이터를 5대 전역 키에 일괄 반영한다.
- 파이프라인 상태는 `_gp89_get_pipeline_status()` 함수로 언제든지 조회 가능하다 (scan/search/script/send 4단계).
- **구현**: `_GP89_PIPE` 상수, `_gp89_set_customer()`, `_gp89_get_pipeline_status()`, `_gp90_sync()` 함수

### §2 1인칭 화법 전역 연동 (Universal First-Person Tone — GP80 Enforcement)
>
- 앱 내 **모든 텍스트 블록**(스캔 리포트, 카카오 미리보기, 검색 결과, 상담 이력)에서 2·3인칭 표현을 1인칭으로 자동 변환한다.
- 금지 표현 → 대체 표현 매핑 (`_GP89_TONE_MAP`):
  - `당신` → `나`, `고객` → `나`, `귀하` → `나`, `피보험자` → `나`, `계약자` → `나`
  - `필요하십니다` → `필요합니다`, `하실 수 있습니다` → `할 수 있습니다`
  - `드립니다` → `드립니다` (유지), `하세요` → `해야 합니다`
- 변환 함수 `_gp89_tone_filter(text)`는 모든 텍스트 출력 전에 파이프라인 내에서 자동 적용된다.
- **구현**: `_GP89_TONE_MAP` 상수, `_gp89_tone_filter()` 함수
>
### §3 카카오톡 허브 상담 이력 자동 저장 (KakaoTalk Hub Auto-History)
>
- **발송 완료 버튼** 클릭 시 `_gp89_save_history()` 함수가 자동 호출되어 `consultation_history` 세션 키에 기록한다.
- 저장 레코드 구조:
  - `일자`: 발송 시각 (YYYY-MM-DD HH:MM)
  - `고객명`: `_gp90_name` 또는 `_gp89_customer_name`
  - `플랜명`: `_gp89_customer_plan`
  - `사용된 문구`: `_gp88_active_script` 앞 60자
  - `OCR 담보`: `_gp88_ocr_result` 핵심 3건
  - `발송 메시지`: 전송 메시지 앞 80자
- `KAKAO_JS_KEY` 설정 여부와 무관하게 **두 경로 모두** 이력 저장을 제공한다 (SDK 발송 / 텍스트 복사 폴백).
- 이력 최대 보존 건수: **200건** (초과 시 오래된 것부터 자동 삭제).
- **구현**: `_gp89_save_history()` 함수, `_gp88_kakao_send()` 내 저장 버튼
>
### §4 시각적 유기성 — 4단계 흐름 배너 (Visual Pipeline Flow Banner)
>
- `_gp88_hub()` 진입 시 **4단계 파이프라인 진행 상태 배너**를 자동 표시한다.
- 각 단계: `📄 계약서 스캔` → `🔍 고객·상담 검색` → `✍️ 화법·문구 선택` → `📲 카카오·증서 발송`
- 완료된 단계: 초록 배경 + ✅ 체크마크 / 미완료 단계: 회색 배경 + 번호 표시
- 활성 고객명이 존재하면 배너 하단에 `👤 현재 고객: [이름]` 배지를 추가 표시한다.
- **구현**: `_gp89_flow_banner()` 함수
>
### §5 GP90 전역 고객 데이터 패널 (Global Customer Data Panel)
>
- **GP90 패널** (`_gp90_panel()`)은 앱 최상단에 상시 노출하여 성명·주민번호·연락처·계약사항·증권분석을 단일 입력창에서 받는다.

- 주민번호는 입력 즉시 **마스킹 처리** (뒷자리 `*` 6개로 대체), 만 나이·성별 자동 계산.
- 입력값 기반 **실시간 과거 상담 검색** (Auto-fill 제안): 2자 이상 입력 시 `_gp90_live_search()` 자동 실행.
- 계약·분석 데이터 결합으로 **개인화 1인칭 화법 문구** 자동 생성 (`_gp90_rebuild_persona()`).
- **구현**: `_GP90_KEYS` 상수, `_gp90_sync()`, `_gp90_parse_rrn()`, `_gp90_rebuild_persona()`, `_gp90_live_search()`, `_gp90_panel()` 함수
>
### §6 시스템 통합 원칙
>
```

       ↓ 전역 동기화 (_gp90_sync)
[계약서 OCR 스캔] → [상담 검색] → [화법·문구 선택] → [카카오 통합 발송 + 증서 발급]
```

## 제91조 1인칭 실시간 생애주기 리스크 팝업 규정 (Intelligent Lifecycle Risk Modal — GP91)
> **신설: 2026-03-08**
> 본 조항은 주민번호 13자리 입력 즉시 고객의 생애주기를 진단하고, 가장 아픈 리스크를 1인칭 고백체로 출력하는 '인텔리전트 팝업'의 표준을 정의한다.
> "입력과 동시에 상담의 정점에 도달하라. 이것이 마스터의 디지털 영업 방식이다."

---

### §1 동적 연령 엔진 (Dynamic Age Engine)

- **주민번호 13자리** 입력 완성 즉시 `_gp90_parse_rrn()` → `_gp91_trigger()` 자동 호출 (GP90 패널 동기화 버튼 경유).
- 만나이 계산 공식: `현재연도 - 출생연도 - (오늘이 생일 이전이면 1)`.
- 4대 생애주기 구간 매칭 (`_GP91_LIFECYCLE`):
  - 🌱 **자산형성기** (20~34세): 암·질병 대비 황금기
  - 🏠 **자녀교육기** (35~49세): 가장 책임 최고조, 교육비 준비
  - 🌅 **은퇴준비기** (50~64세): 노후 독립·의료비·상속 준비
  - 👑 **자산전수기** (65세+): 상속세 재원 확보, 가업 승계
- 해당 구간 없으면 리포트 미표시 (정상 종료).
- **구현**: `_gp91_get_lifecycle()`, `_gp91_trigger()`, `_gp91_build_report_key()` 함수

### §2 1인칭 리스크 리포트 콘텐츠 규정 (Content Standard)

- 각 생애주기별 **3대 핵심 리스크** 문장은 반드시 `나`를 주어로 작성한다 (GP80·GP85 준수).
- 각 리스크 항목은 **구체적 통계 수치** 포함 필수:
  - 자산형성기: 평균 치료비 3,800만원
  - 자녀교육기: 대학 4년 비용 평균 1억2천만원, 40대 남성 사망률 최고
  - 은퇴준비기: 한국인 평균 수명 84세, 은퇴 후 의료비 1억5천만원
  - 자산전수기: 10억 자산 상속세 최대 50%(2억4천만원)
- **1인칭 독백(Monologue)** 블록: 이탈릭체 인용구 형식, 설계사가 고객의 내면을 대신 고백하는 화법.
- **3대 액션 항목**: `오늘 내가 확인해야 할 항목` — 구체적 수치와 함께 즉시 실행 가능한 내용.
- **구현**: `_GP91_LIFECYCLE` 상수 (생애주기별 `label/range/icon/color/desc/stat/monologue/risks/actions`)

### §3 황금빛 Gold-leaf 모달 UI 규정 (Visual Design Standard)

- `st.components.v1.html()`로 렌더링하는 완전 커스텀 HTML/CSS 팝업.
- **외부 테두리**: `box-shadow: 0 0 0 3px [생애주기 색상], 0 0 0 6px [색상]40` — Gold-leaf 이중 링 효과.
- **상단 크라운 배지**: 시머링 그라데이션 애니메이션 (`@keyframes gp91Shimmer`) 적용, 절대 위치.
- **컬러 시스템**: 생애주기별 고유 색상 (자산형성기 #0ea5e9 / 자녀교육기 #f59e0b / 은퇴준비기 #8b5cf6 / 자산전수기 #D4AF37).
- **팝업 높이**: 680px, 스크롤 허용 (`scrolling=True`).
- **페이드인 애니메이션**: `@keyframes gp91FadeIn` — `translateY(-22px) scale(0.96)` → 정위치.
- **구현**: `_gp91_report_panel()` 함수

### §4 전역 데이터 연동 규정 (Global Integration)

- 리포트 하단 **4개 버튼**:
  1. 🔑 **황금빛 열쇠 확인하기** (Primary 버튼): 리스크 요약 → `_gp90_analysis`, `_gp89_active_script`, `consultation_history` **3개 키 동시 반영**
  2. 📋 **상담 기록에 저장**: `_gp89_save_history()` 호출
  3. 📜 **증서 기초데이터로 설정**: `_gp90_analysis` + `_gp89_active_script` 갱신
  4. ✖ **닫기**: 팝업 플래그 `_gp91_report_visible = False` → `st.rerun()`
- 리포트 생성 시 `_gp91_risk_summary` 세션 키에 전문 저장 → GP88 OCR 탭, GP86 증서에서 자동 참조 가능.
- `_gp90_analysis`가 이미 입력된 경우 덮어쓰지 않고 유지 (트리거 시 `if not` 조건 적용).
- **호출 위치**: `_gp90_panel()` 직후, `_gp88_hub()` 직전 — 상담 결과 출력 영역 내 자동 렌더링.
- **구현**: `_gp91_trigger()` 내 `_gp91_risk_summary` 저장, `_gp91_report_panel()` 내 버튼 로직

### §5 표시 조건 및 라이프사이클

- **표시 조건**: `st.session_state["_gp91_report_visible"] == True`
- **자동 트리거**: GP90 패널에서 동기화 버튼 클릭 → `_gp90_sync()` → `_gp91_trigger()` 자동 호출
- **중복 방지**: 이미 `_gp90_analysis`에 값이 있으면 분석 키 덮어쓰기 생략
- **닫기 후 재진입**: 새 주민번호 입력 시 플래그 재설정으로 리포트 재표시

### §6 베테랑 영업 철학 적용

```

       ↓ (입력 즉시)

```

- 고객은 자신의 미래를 보여주는 거울 앞에 서게 되며 **"귀하게 대접받는 느낌"**을 받는다.
- 설계사는 분석 도구를 조작하는 것이 아닌 **30년 경험이 응축된 AI 리포트를 함께 검토**한다.

## 제92조 마스터의 애민정신 육성 코너 규정 (Master's Empathy Voice Corner — GP92 Lifecycle)
> **신설: 2026-03-08**

## 제93조 마스터의 육성 진심 연동 엔진 (Master's Voice Analog Engine — GP93)
> **신설: 2026-03-09**
> 본 조항은 GoldKey AI 마스터가 자신의 목소리로 고객에게 약속을 전달하는 '육성 진심 엔진'의 표준을 정의한다.
> "디지털 증서에 마스터의 목소리를 새겨라. 글자는 지워지지만, 목소리는 마음에 영원히 남는다."

### §1 마스터 전용 음성 레코더 (Master Voice Recorder)

- **위치**: `_gp86_golden_card()` 증서 발급 완료 직후, 카카오톡 공유 버튼 상단에 배치.
- **최대 녹음 시간**: 30초 (자동 종료).
- **구현 방식**: 브라우저 `MediaRecorder API` (WebM/Opus 코덱) 기반 인라인 컴포넌트.
- **파형 애니메이션**: 녹음 중 황금빛 8-bar 파형(`gp93-wave`) 실시간 표시.
- **타이머**: `00:00` 형식 카운트업, 30초 도달 시 자동 녹음 종료.
- **GCS 업로드**: 녹음 완료 후 `☁️ GCS에 업로드` 버튼 → `_gp93_upload_voice()` 호출.
- **파일명 규칙**: `master_voice/{customer_name}_{YYYYMMDD}_{uuid4}.webm`
- **폴백**: 브라우저 미지원 환경을 위해 `📁 파일 직접 업로드` 확장 패널 제공.
- **구현 함수**: `_gp93_voice_recorder(customer_name, plan, date_str)`
- **세션 키**: `_gp93_voice_url` (공개 GCS URL), `_gp93_voice_bytes` (원시 바이트)

### §2 카카오톡 공유 확장 — 육성 청취 버튼 (KakaoTalk Voice Button)

- GP87 `_gp87_kakao_share()` 함수는 세션 키 `_gp93_voice_url` 존재 여부를 조회한다.
- URL이 존재하면 카카오 피드 메시지의 `buttons` 배열에 다음 버튼을 **자동 추가**:
  ```
  title: '🎧 마스터의 진심 목소리 듣기'
  link.webUrl / link.mobileWebUrl: <GCS 공개 URL>
  ```
- URL이 없으면 버튼 미추가 (기존 `📜 약속 확인하기` 버튼만 출력).
- `KAKAO_JS_KEY` 미설정 시 텍스트 복사 모드에서도 `_voice_url`이 자동 포함된다.
- **구현 위치**: `_gp87_kakao_share()` 내 `_voice_btn_js` 조건 분기

### §3 황금빛 오디오 플레이어 페이지 (Golden Audio Player)

- 음성 URL이 세션에 저장되면 `_gp93_voice_recorder()` 내 `🎧 황금빛 오디오 플레이어 미리보기` 확장 패널에 렌더링된다.
- **배경**: `linear-gradient(135deg, #1a1200, #2d1f00, #1a1200)` — 딥 다크 골드 그라디언트.
- **헤더 배지**: "🎙️ Master's Voice" + 황금빛 shimmer 애니메이션.
- **프로필 원형 아이콘**: `🔑` 인장, 황금 그라디언트 테두리 + shimmer.
- **약속 카드**: 1인칭 메시지 + 고객명 + 플랜명 + 날짜 표시.
- **오디오 플레이어**: `<audio controls>` + `accent-color: #D4AF37` 황금 스킨.
- **파형 애니메이션**: 재생 중 15-bar 황금 파형 (`gp93-player-wave`) CSS 애니메이션.
  - 재생(play) 이벤트 → `.active` 클래스 추가 → 파형 시작.
  - 일시정지/종료(pause/ended) → `.active` 제거 → 파형 정지.

## 제95조 [첨단 암 담보 전문 지식 내재화 엔진 (Advanced Cancer Rider Intelligence)]

### §1 목적 및 적용 범위

첨단 암 치료(CAR-T 세포 치료·중입자 치료·양성자 치료·표적 치료)에 대한 급여/비급여 하이브리드 담보 구조를 시스템 내에 내재화하여, 상담사가 복잡한 의료비 구조를 1인칭 화법으로 즉시 설명할 수 있도록 한다.

### §2 첨단 암 담보 데이터 구조 (`_GP95_ADVANCED_RIDERS`)

각 담보 항목은 다음 필드를 보유한다:

| 필드 | 설명 |
|------|------|
| `name` | 담보명 (한글) |
| `category` | 분류 (CAR-T / 중입자 / 양성자 / 표적 치료 등) |
| `insured_cost` | 급여 본인부담금 범위 |
| `non_insured_cost` | 비급여 비용 범위 |
| `coverage_limit` | 보험 지급 한도 |
| `hook` | 1인칭 화법 핵심 후킹 문구 |
| `script` | 상담 스크립트 전문 |

### §3 하이브리드 시나리오 (`_GP95_HYBRID_SCENARIO`)

급여 + 비급여 비용이 동시에 발생하는 현실적 시나리오를 정의한다.

- **시나리오**: 급성 백혈병 환자가 CAR-T 세포 치료 + 중입자 방사선 병행 수술을 받는 경우
- **총 예상 비용**: 약 2억 5천만 원 이상
- **급여 본인부담**: 약 500만 원 (산정특례 5%)
- **비급여 비용**: CAR-T 1.5억 원 + 중입자 방사선 6,000만 원 = 약 2.1억 원
- **보험 지급 시나리오**: 첨단 암 치료 특약(급여 본인부담금 환급) + 비급여 항암 특약(비급여 직접 보전)
- **1인칭 후킹**: "국내 최첨단 치료를 받는 날, 나는 돈 걱정 없이 오직 치료에만 집중할 수 있어야 합니다."

### §4 담보 비교표 (`_GP95_COVERAGE_COMPARISON`)

주요 손해보험사 첨단 암 특약을 항목별로 비교한다:

| 항목 | 현대해상 | DB손보 | KB손보 | 삼성화재 | 메리츠화재 |
|------|---------|--------|--------|---------|-----------|
| CAR-T 세포 치료 | 급여 본인부담금 환급 | 급여+비급여 혼합 | 비급여 특약 별도 | 급여 본인부담금만 | 비급여 항암 포함 |
| 중입자 치료 | 별도 특약 | 포함 | 포함 | 별도 | 포함 |
| 양성자 치료 | 포함 | 포함 | 별도 | 포함 | 포함 |
| 표적 치료제 | 일부 포함 | 포함 | 포함 | 일부 | 포함 |

### §5 UI 패널 (`_gp95_panel()`)

- **탭 구조**: 「담보 상세」 / 「하이브리드 시나리오」 / 「회사별 비교」
- **스크립트 다운로드**: 선택한 담보의 1인칭 상담 스크립트를 `.txt`로 다운로드
- **GP88 연동**: `st.session_state["_gp88_selected_script"]`에 스크립트 자동 추가
- **1인칭 화법 강제**: 모든 후킹·스크립트 문구는 GP80 1인칭 화법 표준 준수

### §6 1인칭 화법 준수 원칙

- ✅ 허용: `"나는 오늘 이 치료를 받을 권리가 있습니다."`
- ❌ 금지: `"고객님의 치료를 위해"`, `"귀하의 보장"`
- 모든 후킹 문구는 `hook` 필드에서 관리하며 수정 시 GP80 검증 로직 통과 필수

## 제96조 [리플렛 자산 허브 및 RAG 검색 체계 (Leaflet Asset Hub with RAG)]

### §1 목적 및 적용 범위

보험사 상품 리플렛(PDF/이미지)을 체계적으로 수집·분류·저장하고, 상담 중 자연어 질의로 즉시 검색할 수 있는 RAG(Retrieval-Augmented Generation) 체계를 구축한다.

### §2 GCS 저장 구조

```
insu-archive-2026/
└── leaflets/
    └── {company}/
        └── {product_type}/
            └── {YYYYMM}/
                └── {uuid}_{filename}
```

- **버킷**: `insu-archive-2026` (기존 버킷 재사용)
- **Prefix**: `leaflets/`
- **경로 분류 기준**: 회사명 → 상품 유형 → 연월

### §3 OCR 메타데이터 자동 추출 (`_gp96_extract_meta()`)

업로드된 파일(PDF/이미지)에서 다음 메타데이터를 자동 추출한다:

| 필드 | 추출 방법 | 예시 |
|------|----------|------|
| `company` | 키워드 매칭 | "현대해상", "DB손보" |
| `product` | 키워드 매칭 | "암보험", "실손보험" |
| `riders` | 키워드 매칭 리스트 | ["CAR-T", "중입자", "양성자"] |
| `date` | 정규식 (YYYYMMDD) | "20250601" |
| `raw_text` | OCR 전체 텍스트 | — |

- **PDF 추출**: `pdfplumber` 사용 (설치 시) / 폴백: 텍스트 디코딩
- **이미지 추출**: `pytesseract` 사용 (설치 시) / 폴백: 파일명 기반 추론
- **키워드 사전**: 회사 15개, 상품 유형 10개, 특약/담보 20개 이상

### §4 세션 인덱스 (`_gp96_leaflet_index`)

`st.session_state["_gp96_leaflet_index"]`에 누적 저장되는 인덱스 구조:

```python
{
    "company": str,
    "product": str,
    "riders": list[str],
    "date": str,
    "gcs_url": str,
    "raw_text": str,
    "saved_at": str   # ISO 타임스탬프
}
```

### §5 RAG 검색 엔진 (`_gp96_rag_search()`)

- **검색 방식**: 키워드 토큰화 후 인덱스 내 `raw_text` 전문 매칭 + 메타데이터 필터
- **랭킹**: 매칭 키워드 수 기준 내림차순
- **결과**: 최대 5건, 각 결과에 GCS URL / 회사명 / 담보 목록 표시
- **자연어 예시**: "요즘 중입자 치료 얼마 들어?" → `중입자` 키워드 추출 → 관련 리플렛 검색

### §6 UI 패널 (`_gp96_panel()`)

- **탭 구조**: 「리플렛 업로드」 / 「RAG 검색」 / 「저장 인덱스」
- **업로드 흐름**: 파일 업로드 → OCR 자동 추출 → 메타데이터 표시 → GCS 저장 버튼
- **수동 입력**: 메타데이터 미추출 시 수동 오버라이드 가능
- **GP97 자동 트리거**: GCS 저장 완료 시 `_gp97_analyze_diff()` 자동 호출

### §7 GP97 자동 연동 규칙

- 인덱스에 2건 이상 누적된 시점에서 신규 저장 시 **반드시** GP97 차이 분석을 트리거한다.
- 신규 담보가 감지되면 `st.info()`로 즉시 사용자에게 알린다.
- 트리거는 「GCS에 저장」 버튼과 「인덱스만 등록」 버튼 모두에서 작동한다.

## 제97조 [미래 상품 자율 학습 지침 (Autonomous Product Learning)]

### §1 목적 및 적용 범위

신규 리플렛 등록 시 기존 인덱스와의 담보 차이를 자동 분석하여 신규 트렌드를 등록하고, 상담 스크립트를 자동 갱신하는 자율 학습 체계를 구축한다.

신규 리플렛의 `riders` 목록과 기존 인덱스 전체의 `riders` 합집합을 비교하여 추가/삭제된 담보를 반환한다.

```python
def _gp97_analyze_diff(new_meta: dict, existing_index: list) -> dict:
    # 반환 형태
    return {
        "added": list[str],    # 신규 등장 담보
        "removed": list[str],  # 소멸/삭제 담보
        "company": str,
        "product": str,
        "date": str
    }
```

### §3 트렌드 등록 (`_gp97_register_trend()`)

`st.session_state["_gp97_trends"]`에 트렌드를 누적 저장한다:

```python
{
    "added": list[str],
    "removed": list[str],
    "company": str,
    "product": str,
    "date": str,
    "detected_at": str   # ISO 타임스탬프
}
```

- 동일 담보의 중복 등록을 방지한다 (`added` 항목 기준 dedup).

### §4 상담 문구 자동 갱신 (`_gp97_get_latest_script_update()`)

트렌드 목록을 기반으로 최신 1인칭 상담 후킹 문구를 자동 생성한다:

```
"{신규 담보A}와 {신규 담보B}가 시장에 새롭게 등장했습니다.
나는 이 최신 트렌드를 가장 먼저 고객에게 전달할 수 있는 전문가입니다."
```

- 트렌드가 없으면 기존 GP95 스크립트를 그대로 활용한다.
- GP88 `_gp88_selected_script` 세션 키에 자동 추가 버튼 제공.

### §5 UI 패널 (`_gp97_panel()`)

- **탭 구조**: 「트렌드 현황」 / 「자동 갱신 문구」
- **트렌드 테이블**: 감지 일시 / 회사 / 상품 / 추가 담보 / 삭제 담보
- **자동 갱신 문구**: 최신 트렌드 기반 스크립트 표시 + GP88 추가 버튼
- **수동 클리어**: 트렌드 목록 초기화 버튼

### §6 자율 학습 원칙

1. **자동성**: 사용자가 별도 조작 없이 리플렛 저장 시 자동 실행된다.
2. **투명성**: 감지된 트렌드는 반드시 UI에 즉시 표시한다.
3. **1인칭 화법**: 자동 생성 스크립트는 GP80 1인칭 화법 표준을 준수한다.
4. **비파괴성**: 기존 트렌드 데이터를 덮어쓰지 않고 누적 저장한다.
5. **연동성**: GP95(담보 DB), GP96(리플렛 인덱스), GP88(상담 허브)와 완전 연동된다.

### §7 시스템 통합 흐름

```
[GP96: 리플렛 업로드 → OCR → GCS 저장]
       ↓ (저장 완료 시 자동 트리거)
[GP97: _gp97_analyze_diff() — 담보 차이 분석]
       ↓
[GP97: _gp97_register_trend() — 트렌드 누적 등록]
       ↓
[GP97: _gp97_get_latest_script_update() — 1인칭 스크립트 생성]
       ↓
[GP88: _gp88_selected_script — 상담 허브 자동 추가]
       ↓
[GP95: _GP95_ADVANCED_RIDERS — 첨단 담보 DB와 교차 검증]
```

## 제130조 [데이터 보호 특칙 — 의료 데이터 보안 및 개인정보 처리방침]

> 이 조항은 골드키 앱의 **이용약관 및 개인정보 처리방침의 최상위 법적 효력을 갖는 데이터 보호 특칙**이다.  
> AI 및 개발자는 의료 데이터를 처리하는 모든 코드에서 이 조항을 최우선 원칙으로 적용한다.  
> **시행일: 2026년 3월 8일**

---

### §1 [자동 비식별화 원칙 — 수집 단계 마스킹 의무]

회사는 이용자가 업로드하는 **의무기록 및 진단서의 민감정보**를 보호하기 위해, 수집 단계에서 **성명 및 주민등록번호 뒷자리를 자동 비식별화(Masking)** 처리한다.

- **적용 대상:** 의무기록, 진단서, 수술확인서, 처방전, 입·퇴원 확인서 등 업로드되는 모든 의료 문서
- **마스킹 범위:**
  - 성명: 성(姓)을 제외한 나머지 글자를 `*`로 대체 (예: 이** )
  - 주민등록번호 뒷자리: 전체 7자리를 `*******`로 대체
  - 연락처: 중간 4자리를 `****`로 대체
- **처리 시점:** 파일 업로드 즉시(서버 도달 전) 클라이언트 측 1차 마스킹 + 서버 측 2차 마스킹 이중 적용
- **구현 함수:** `_gp120_mask_pii()` (app.py GP120 블록)

---

### §2 [AES-256 암호화 원칙 — 저장 및 전송 보안]

모든 의료 데이터는 **국제 표준인 AES-256(Advanced Encryption Standard, 256-bit) 방식**으로 암호화되어 저장 및 전송된다.

- **암호화 적용 범위:**
  - 업로드된 의료 문서 원본 파일 전체
  - 비식별화 처리 후 남은 의료 텍스트 데이터
  - OCR 추출 결과물 (KCD 코드, 진단명, 수술명 포함)
  - GP100/GP110/GP120 처리 중간 결과물
- **키 관리:** 암호화 키는 서버 측 환경 변수로만 관리하며, 코드 내 하드코딩 절대 금지
- **전송 보안:** 모든 데이터 전송은 TLS 1.2 이상 적용
- **구현 함수:** `_gp120_encrypt_data()`, `_gp120_decrypt_data()` (app.py GP120 블록)

---

### §3 [GCS 보안 버킷 격리 저장 및 접근 통제 원칙]

암호화된 의료 데이터는 **외부와 격리된 Google Cloud Storage(GCS) 보안 버킷**에 독립 저장되며, 엄격한 접근 권한 통제를 통해 **마스터 본인 외에는 누구도 열람할 수 없음**을 보장한다.

- **저장 위치:** GCS 전용 보안 버킷 (공개 접근 차단, 서비스 계정 기반 IAM 인증 전용)
- **접근 통제 체계:**
  - IAM(Identity and Access Management) 기반 역할 제한
  - 마스터(설계사 본인) 계정 외 읽기·쓰기 권한 일체 차단
  - 관리자(Admin) 계정도 원본 데이터 직접 열람 불가 (암호화된 상태로만 저장됨)
- **감사 로그:** 모든 데이터 접근·다운로드·삭제 이벤트는 `GP120_VAULT_LOG`에 타임스탬프와 함께 자동 기록
- **데이터 보존 기간:** 마스터가 직접 삭제 요청 시 즉시 영구 삭제 (디지털 소각 — GCS `delete()` + 로컬 캐시 `del` 동시 실행)
- **구현 함수:** `_gp120_save_to_vault()`, `_gp120_destroy_data()` (app.py GP120 블록)

---

### §4 [데이터 보호 특칙 준수 확인 체크리스트]

| 항목 | 구현 상태 | 구현 위치 |
|------|-----------|-----------|
| 성명 자동 마스킹 | ✅ 구현 완료 | `_gp120_mask_pii()` |
| 주민등록번호 뒷자리 자동 마스킹 | ✅ 구현 완료 | `_gp120_mask_pii()` |
| AES-256 암호화 저장 | ✅ 구현 완료 | `_gp120_encrypt_data()` |
| GCS 보안 버킷 독립 저장 | ✅ 구현 완료 | `_gp120_save_to_vault()` |
| IAM 접근 권한 통제 | ✅ 구현 완료 | GCS IAM 정책 |
| 감사 로그 자동 기록 | ✅ 구현 완료 | `GP120_VAULT_LOG` |
| 디지털 소각(영구 삭제) | ✅ 구현 완료 | `_gp120_destroy_data()` |
| 로그인 화면 약관 고지 | ✅ 구현 완료 | `app.py` 라인 ~21227 |

---

### §5 [이용자 권리 보장]

- **열람권:** 마스터는 언제든지 저장된 자신의 의료 데이터 목록을 `나의 인생 방어 사령부 > 골든 볼트` 패널에서 확인할 수 있다.
- **삭제권:** 마스터는 개별 데이터 또는 전체 데이터를 즉시 영구 삭제(디지털 소각)할 수 있다.
- **이의 제기권:** 데이터 처리에 이의가 있을 경우 운영자(goldkey_Ai_masters2026)에게 즉시 신고할 수 있다.
- **처리 거부권:** 민감정보 AI 분석 동의를 철회하면 해당 데이터의 AI 분석이 즉시 중단된다.

---

### §6 [위반 시 대응 원칙]

- 이 조항을 위반하는 코드 변경은 가이딩 프로토콜 최상위 위반으로 간주하며, 즉시 롤백한다.
- 데이터 유출 의심 시 해당 버킷을 즉시 격리하고 마스터에게 인앱 알림을 발송한다.
- 보안 취약점 발견 시 `AUDIT_LOG.md`에 🔴 긴급 등급으로 기록하고 24시간 내 패치를 완료한다.

## 제100조 [GP100 — 보험금 청구 지능(Claim Shield) 프로토콜]

> 이 조항은 **GP100 보상지능 시스템**의 설계 원칙 및 운용 규칙을 정의한다.  
> 마스터의 보험금 청구권을 최대한 방어하고, 보험사의 부당 지급거절에 대한 전략적 대응력을 부여하는 것이 목적이다.  
> **시행일: 2026년 3월 8일**

---

### §1 [목적 및 적용 범위]

- **목적:** 마스터가 업로드하는 의무기록·진단서·수술확인서를 AI로 분석하여, 보험사의 부당 거절 전술을 사전 탐지하고 최적의 대응 논리를 자동 생성한다.
- **적용 범위:** KCD-10 질병코드 기반 보험금 청구 전 과정(청구 준비 → 접수 → 이의신청 → 분쟁 조정)

### §2 [핵심 기능 명세]

| 기능 | 설명 | 구현 키 |
|------|------|---------|
| KCD 방어 DB | 29개 핵심 질병코드별 보험사 거절 전술 + 마스터 반박 논리 | `_GP100_KCD_DEFENSE` |
| 거절 전술 자동 탐지 | 진단서 텍스트에서 거절 위험 신호 자동 감지 | `_gp100_detect_denial_risk()` |
| 대응 논리 생성 | KCD 코드 기반 최적 반박 스크립트 자동 생성 | `_gp100_build_defense()` |
| 황금빛 증서 연동 | 분석 결과를 GP86 황금빛 약속 증서에 자동 통합 | `_gp100_last_defense` |
| 1인칭 화법 | 모든 출력은 GP80 1인칭 화법 표준 준수 | GP80 연동 |

### §3 [운용 원칙]

1. **사전 방어 우선:** 거절 통보를 받기 전, 청구 단계에서 선제적으로 위험을 식별한다.
2. **데이터 기반:** 감정적 호소가 아닌 KCD 코드·약관 조항·판례 근거 중심의 논리를 구성한다.
3. **1인칭 권리 강조:** "나의 권리", "내가 납부한 보험료"를 전면에 내세운 문구를 사용한다.
4. **GP110 연동:** 법리 분쟁 단계로 에스컬레이션 시 GP110 법률RAG에 자동 인계한다.
5. **GP120 보안:** 처리된 의무기록은 즉시 GP120 골든볼트에 암호화 보관한다.

### §4 [황금빛 약속 증서 연동 규칙]

- GP100 분석 완료 시 `_gp100_last_defense` 세션 키에 방어 논리가 저장된다.
- GP86 황금빛 약속 증서 생성 시 이 값을 자동으로 "청구 방어 근거" 항목에 통합한다.
- 분석 결과가 없는 경우 해당 항목을 공란으로 두되, 사용자에게 GP100 분석을 권장한다.

## 제110조 [GP110 — 법률 지능 RAG(Legal Intelligence) 프로토콜]

> 이 조항은 **GP110 법률지능 시스템**의 설계 원칙 및 운용 규칙을 정의한다.  
> 보험 분쟁·의료소송·손해배상 관련 판례를 실시간으로 검색·분석하여, 마스터에게 법적 승소 전략을 제공하는 것이 목적이다.  
> **시행일: 2026년 3월 8일**

---

### §1 [목적 및 적용 범위]

- **목적:** 금융감독원 분쟁조정·법원 판례·보험업법 조문을 RAG(검색증강생성) 방식으로 검색하여, 마스터의 분쟁 상황에 최적화된 법적 대응 전략을 자동 생성한다.
- **적용 범위:** 보험금 지급 거절 이의신청, 금감원 분쟁조정 신청, 민사소송 준비 단계

### §2 [핵심 기능 명세]

| 기능 | 설명 | 구현 키 |
|------|------|---------|
| 판례 RAG DB | 20개 핵심 보험 분쟁 판례 벡터 데이터베이스 | `_GP110_CASES` |
| 키워드 유사도 매칭 | 마스터 상황 입력 → 관련 판례 자동 매칭 | `_gp110_rag_search()` |
| 법적 전략 생성 | 매칭 판례 기반 승소 전략 자동 시뮬레이션 | `_gp110_simulate_strategy()` |
| 분쟁조정 신청서 | 금감원 신청 요건·서류·절차 자동 안내 | `_GP110_DISPUTE_PROC` |
| 황금빛 증서 연동 | 판례·전략을 GP86 증서에 자동 통합 | `_gp110_hits` |

### §3 [RAG 검색 원칙]

1. **관련성 우선:** 키워드 유사도가 2점 이상인 판례만 결과로 표시한다.
2. **최신성 우선:** 동일 유사도 점수 시 최신 판결 연도 판례를 우선 표시한다.
3. **승소율 명시:** 각 판례에 마스터 유리 여부(`favor`)와 보상 배율을 반드시 표시한다.
4. **GP100 인계:** 보험금 거절 관련 판례 매칭 시 GP100 분석 결과와 교차 참조한다.
5. **1인칭 화법:** 모든 전략 문구는 "나의 사건", "내가 받아야 할 보상"을 주어로 사용한다.

### §4 [마스터의 한마디 연동 규칙]

- GP110 분쟁 전략 시뮬레이션 완료 시 `_gp110_sim` 세션 키에 결과가 저장된다.
- GP80 "마스터의 한마디" 생성 시 이 값을 자동으로 "법적 근거" 항목에 통합한다.
- 판례 매칭이 없는 경우에도 분쟁조정 신청 절차 안내를 반드시 포함한다.

## 제120조 [GP120 — 골든 볼트(Golden Vault) 보안 금고 프로토콜]

> 이 조항은 **GP120 골든볼트 시스템**의 설계 원칙 및 운용 규칙을 정의한다.  
> 마스터의 의료 데이터를 철통같이 보호하는 아이언클래드(Ironclad) 보안 체계를 구현하는 것이 목적이다.  
> **제130조 데이터 보호 특칙과 완전 연동되며, 제130조가 GP120의 상위 법적 근거이다.**  
> **시행일: 2026년 3월 8일**

---

### §1 [목적 및 적용 범위]

- **목적:** 업로드된 의무기록·진단서·수술확인서를 비식별화 → 암호화 → 격리 저장하는 3단계 보안 파이프라인을 자동 실행하여, 마스터의 의료 데이터를 완전히 보호한다.
- **적용 범위:** GP100·GP110에서 처리된 모든 의료 데이터, 마스터가 직접 업로드한 의료 문서 전체

### §2 [3단계 보안 파이프라인]

```
[1단계: 자동 비식별화]
  _gp120_mask_pii() — 성명·주민번호 뒷자리·연락처 즉시 마스킹
       ↓
[2단계: AES-256 암호화]
  _gp120_encrypt_data() — 비식별화된 데이터 전체 암호화
  키: 환경변수 GK_VAULT_KEY (하드코딩 절대 금지)
       ↓
[3단계: GCS 보안 버킷 격리 저장]
  _gp120_save_to_vault() — 암호화 데이터 → GCS 전용 버킷 저장
  IAM: 마스터 서비스 계정 외 접근 전면 차단
```

### §3 [핵심 기능 명세]

## [SECTION: PERFORMANCE & STABILITY PROTOCOL] — 제130.1조

> **목표:** Cold Start(첫 구동) 4초 이내 / Warm Start(기능 전환) 1.2초 이내  
> 이 섹션의 모든 규칙은 마스터의 실전 상담 리듬을 방해하지 않도록 **상시 준수**해야 한다.

### §1 Caching First (캐싱 우선)
- GCS 클라이언트 연결은 `@st.cache_resource`로 **프로세스당 1회만** 인증·생성한다. (`_get_gcs_client_cached()`)
- 대용량 데이터·AI 모델 호출 결과는 `@st.cache_data(ttl=3600)`으로 메모리에 상주시킨다.
- 회원 목록(`load_members`)은 세션 캐시 → 프로세스 캐시(TTL 300초) → Supabase → /tmp 순서로 조회한다.
- `save_members()` 호출 직후에는 반드시 `st.session_state.pop("_members_cache", None)`으로 세션 캐시를 무효화한다.

### §2 Lazy Loading (지연 로딩)
- `pandas`, `PIL`, `cryptography.fernet`, `google.genai`, `ftfy` 등 **무거운 라이브러리는 전역 임포트 절대 금지**.
- 반드시 `_lazy_pd()`, `_lazy_pil_image()`, `_lazy_fernet()`, `_lazy_genai()`, `_lazy_ftfy()` 래퍼를 통해 **기능 호출 시점**에만 로드한다.
- 신규 무거운 패키지 추가 시 동일한 `_lazy_*()` 래퍼 패턴을 적용한다.

### §3 Session State Usage (세션 활용 — 검증 후 로딩)
- 동일 세션 내에서 GCS·Supabase 데이터를 **반복 호출하지 않는다**.
- 한 번 가져온 데이터는 `st.session_state`에 저장하고, `if key in st.session_state:` 검증을 먼저 수행한다.
- CSS 전역 주입(`_gp84_inject_global_css`), JS 성능 로그 주입(`_s40_read_ls_perf_js`), localStorage 복원(`_gp140_restore_from_ls`) 등 **세션 1회성 작업**은 `_injected` 플래그로 중복 실행을 방지한다.

### §4 GCS Singleton (GCS 클라이언트 싱글톤)
- `_get_gcs_client()` 는 내부적으로 `@st.cache_resource` 데코레이터를 적용한 `_get_gcs_client_cached()`에 위임한다.
- **매 rerun마다 OAuth 인증을 반복하는 것은 엄격히 금지**한다. (PERF §1)
- GCS 캐시 읽기(`_gcs_cache_get`) 및 쓰기(`_gcs_cache_put`)도 동일한 싱글톤 클라이언트를 재사용한다.

### §5 Visual Stability (시각적 안정성)
- 결과 출력창은 `st.empty()` 또는 `st.container()`로 **자리를 선점**하여 답변 출력 시 화면이 튀는 현상을 방지한다.
- 데이터 로딩 중에는 `st.spinner()`로 전문성을 강조하는 미니멀 스피너를 노출한다.
- 페이지 전체 리렌더링(`st.rerun()`)은 반드시 필요한 경우에만 호출한다.

### §6 Turbo Goal (속도 목표)
| 구분 | 목표 |
|------|------|
| Cold Start (첫 구동, 라이브러리 포함) | **4초 이내** |
| Warm Start (기능 클릭·페이지 전환) | **1.2초 이내** |
| GCS 캐시 히트 응답 | **0.3초 이내** |

*이상 제130.1조는 골드키 총통합헌법 PERFORMANCE 추록(2026.05)에 공식 수록되며, 신규 기능 구현 시 반드시 이 기준을 만족하는지 검토 후 배포한다.*

## [GP-회원관리 §연락처표준] 연락처 데이터 단일 규격화 원칙 (2026-03-16 신설)

> **핵심 명제: "어떤 형식으로 입력해도, 시스템 내부는 항상 순수 숫자 11자리다."**
> 본 조항은 HQ 앱, CRM 앱, GCS, RAG, OPT 시스템 등 프로젝트 전역에서 연락처(전화번호) 데이터의 저장·비교·인증을 단일 표준으로 통일하는 것을 목적으로 한다.

### §1 [표준 함수 정의 — get_clean_phone()]

모든 연락처 처리는 반드시 `get_clean_phone()` 함수를 최우선으로 거친다.

```python
def get_clean_phone(raw: str) -> str:
    if not raw:
        return ""
    return "".join(filter(str.isdigit, str(raw)))
```

- **구현 위치**: `app.py` 및 `shared_components.py` (양쪽 앱 공유)
- **입력 예시**: `"010-1234-5678"` → `"01012345678"` / `"010 1234 5678"` → `"01012345678"`
- 하이픈(-), 공백, 괄호, 특수문자 등 모두 제거 후 순수 숫자만 반환

### §2 [인증 및 로그인 적용 — ID-100-AUTH]

- 사용자가 입력한 모든 연락처 변수를 SHA-256 해싱하기 직전에 반드시 `get_clean_phone()`을 거친다
- `get_sha256(text)` 함수는 내부적으로 `get_clean_phone()`을 호출하여 정규화 후 해싱
- `encrypt_contact(contact)` 함수도 동일하게 `get_clean_phone()` 선처리 필수
- `decrypt_data(stored_hash, input_data)` 함수도 입력값을 `get_clean_phone()`으로 정규화 후 비교
- 로그인 Track 1~4 모두 정규화된 값으로 검증하여 하이픈 유무에 따른 인증 실패 원천 차단

### §3 [GCS 정보 관리 및 저장 적용 — ID-200-REG]

- 신규 회원 가입 및 정보 수정 시, Supabase/GCS에 저장되기 전 단계에서 `get_clean_phone()` 적용
- 저장되는 해시값(`pin_hash`, `contact`)은 반드시 **순수 숫자 기반** SHA-256 해시여야 한다
- `ensure_master_members()` 함수의 `_pin_hash` 생성도 `get_clean_phone()` 경유 필수

### §4 [전역 일관성 규칙]

| 처리 단계 | 규칙 |
|----------|------|
| 사용자 입력 수신 | `get_clean_phone()` 즉시 적용 |
| SHA-256 해싱 | 정규화된 값만 허용 (`get_sha256()` 사용) |
| DB 저장 (pin_hash) | `get_clean_phone()` 후 해시 저장 |
| 로그인 비교 | `get_sha256()` 통해 정규화 후 비교 |
| Fernet 복호화 비교 | 복호화 결과와 `get_clean_phone(입력값)` 비교 |

### §5 [Legacy 데이터 호환 — Track 4]

하이픈 포함 형식으로 등록된 기존 회원(레거시 데이터) 지원을 위해 Track 4를 운영한다:
- 입력 연락처를 정규화 후 `"XXX-XXXX-XXXX"` 형식으로 복원하여 재해시
- Track 1~3 실패 시에만 Track 4 시도 (HQ 메인/사이드바 로그인 공통 적용)
- CRM 앱도 동일한 `get_clean_phone()` 기반 3-track 검증 적용

### §7 [Silent Update — 레거시 해시 자동 교정]

Track 4 (레거시 하이픈 해시) 로 로그인 성공 시 DB를 신형 규격으로 자동 갱신한다:

- 조건: `_legacy_ok == True` AND `_pin_ok / _contact_hash_ok / _fernet_ok` 모두 False
- 처리: `_silent_update_pin_hash(name, clean_contact)` 즉시 호출
  - `gk_members` 테이블의 `pin_hash`를 `sha256(순수숫자)` 으로 업데이트
  - 성공/실패와 무관하게 로그인 흐름을 중단하지 않음 (Silent)
  - 캐시(`_get_member_cache`, `st.session_state["_members_cache"]`) 즉시 무효화
- 적용 위치: HQ 메인 로그인 (`app.py` ~28798), 사이드바 로그인 (~29272)

### §8 [Data Storage Gateway — 저장 게이트웨이]

모든 회원 데이터 저장 경로의 입구에서 `prepare_member_data(raw_data)` 를 호출하여 연락처를 정규화한다:

- `contact` 필드: 원문이 SHA-256/Fernet 형식이 아닌 경우 `get_clean_phone()` → SHA-256 해시로 치환
- `pin_hash` 필드: 누락된 경우 `get_clean_phone()` → SHA-256 해시로 자동 보완
- 적용 원칙: 신규 가입, 정보 수정, 관리자 변경 등 어떤 경로로 들어오든 DB에는 순수 숫자 기반 해시만 저장

### §9 [DB 쓰기 에러 로깅 원칙]

Supabase 저장 작업 실패 시 조용히 넘어가지 않고 명확한 에러를 출력한다:

- `save_members`, `ensure_master_members`, `_silent_update_pin_hash` 등 모든 DB 쓰기 함수에서 예외 발생 시 `st.error()` 또는 로그 출력 강제
- Silent 허용 예외: `_silent_update_pin_hash` — 로그인 흐름 중단 방지 목적으로만 silent 허용
- 그 외 저장 실패는 반드시 사용자에게 가시적 에러 메시지 노출
