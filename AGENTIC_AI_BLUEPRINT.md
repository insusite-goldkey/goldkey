## 1. 아키텍처 전환 목표 (Context)

### 1.1 현행 시스템 문제점
- **HQ 앱**: 사용자 UI와 AI 연산 로직이 혼재되어 있어 유지보수 어려움
- **CRM 앱**: 단순 데이터 입력 도구로만 활용되며, AI 분석 결과를 실시간으로 확인하기 어려움
- **워크플로우**: 사용자가 단계별로 버튼을 클릭하며 진행해야 하는 문답형 구조

### 1.2 에이전틱 AI 전환 목표

#### HQ 앱 → 중앙 연산 서버 (Backend)
- **역할**: AI 연산(Cloud Run), RAG 지식베이스(Vector DB), 프롬프트 통제, 작업 큐(Job Queue) 관리
- **UI 제거**: 사용자 인터페이스 완전 배제
- **API 전환**: RESTful API 또는 gRPC 엔드포인트로 전환

#### CRM 앱 → 현장 영업용 프론트엔드 (Frontend)
- **상단**: 카테고리 선택 박스 (HQ에서 제공하는 분석 카테고리 목록)
- **하단**: 5:5 스플릿 스크린
  - **좌측 (Input)**: 기초 자료 스캔/입력 (증권, 의무기록, 고객 정보 등)
  - **우측 (Output)**: AI 결과 보고서 출력 (PDF, 요약, 그래프 등)

### 1.3 에이전틱 AI 워크플로우
```
사용자(설계사) → 원시 데이터 업로드 (CRM)
                    ↓
              HQ AI 엔진 자동 처리
              (백그라운드 연산)
                    ↓
              최종 PDF 보고서 반환 (CRM)
```

## 3. 에이전틱 AI 메인 카테고리 재분류 체계

### 3.1 재분류 원칙
1. **사용자 관점**: 설계사가 실제 현장에서 필요한 상담 시나리오 중심
2. **데이터 중심**: 입력 데이터 유형(증권, 의무기록, 재무제표 등)으로 그룹화
3. **AI 연산 복잡도**: 단순 조회 vs 고부하 AI 분석으로 구분
4. **비즈니스 가치**: 코인 차감 기준(0코인 vs 1코인 vs 3코인)

### 4.2 카테고리 1: 보험 증권 정밀 분석

3. **약관 매칭 결과**
   - 가입 시점 약관 자동 추적
   - 약관 변경 이력 (구약관 vs 신약관)
   - 핵심 약관 조항 요약

4. **PDF 보고서**
   - 고객 맞춤형 보장 분석 리포트
   - 가문안보 리포트 (가족 전체 보장 현황)

- [ ] 건강 검진 결과 (최근 1년)
- [ ] 기왕증 이력 (과거 질병/수술)
- [ ] 고지 의무 사항 (보험 가입 시)

#### 우측 (Output) — AI 보고서 종류 및 핵심 도출 값
1. **직업 분류 및 상해 급수 판정**
   - 대분류/중분류/소분류 자동 매칭
   - 상해 급수 (1급~4급)
   - 직업 위험 등급 (고위험/일반)

2. **고위험 질환 분석**
   - KCD 코드 기반 위험도 분석
   - 고위험 질환 배지 (당뇨/고혈압/암 등)
   - 보험 가입 제한 여부

3. **거절 스크립트 생성**
   - 고객 거절 시나리오별 대응 스크립트
   - 심리 전략 (후킹 스크립트)
   - 그라운딩 기법 (공감 + 설득)

4. **소득 역산 연산기**
   - 건강보험료 → 추정 월소득
   - 일일 경제 가치 산출
   - 휴업 손해 계산

### 4.10 카테고리 10: 고객 자료 관리 및 스케줄링

#### 좌측 (Input) — 필수 서류 체크리스트
- [ ] 의무기록 PDF/이미지 업로드
- [ ] 상담 카탈로그 (리플렛, 제안서)
- [ ] 고객 서류 (계약서, 청구서)
- [ ] 일정 정보 (만기일, 생일, 기념일)

#### 우측 (Output) — AI 보고서 종류 및 핵심 도출 값
1. **의무기록 저장 및 관리**
   - GCS 암호화 저장
   - 파일 태깅 (agent_id + person_id)
   - 스캔 이력 조회

2. **상담 카탈로그 열람**
   - 보험사별 리플렛 목록
   - 신상품 자동 등록
   - 카탈로그 AI 분류

3. **간병비 컨설팅**
   - 장기요양 등급 산정
   - 치매보험 설계
   - 요양병원 보장 분석

4. **부동산 투자 분석**
   - 등기부등본 자동 파싱
   - 건축물대장 분석
   - 투자 수익률 계산

5. **건강보험료 역산 계산**
   - 건보료 → 추정 소득
   - 일당 한도 분석
   - 수술비 보완 설계

## 5. HQ/CRM 역할 분리 및 데이터 흐름

### 5.1 HQ 앱 → 중앙 연산 서버 (Backend)

#### 역할
- **AI 연산**: Gemini API, GPT API 호출
- **RAG 시스템**: Vector DB (Pinecone/Weaviate) 연동
- **프롬프트 통제**: 시스템 프롬프트 관리
- **작업 큐**: Celery/Redis 기반 비동기 처리
- **데이터 저장**: Supabase DB 저장

#### API 엔드포인트 (예시)
```
POST /api/v1/analyze/policy
POST /api/v1/analyze/claim
POST /api/v1/analyze/disease
POST /api/v1/analyze/accident
POST /api/v1/analyze/fire
POST /api/v1/analyze/inheritance
POST /api/v1/analyze/ceo
POST /api/v1/analyze/new_insurance
POST /api/v1/analyze/job_risk
POST /api/v1/analyze/customer_data
```

#### 데이터 흐름
```
CRM (사용자 입력)
    ↓
HQ API (POST /api/v1/analyze/*)
    ↓
Job Queue (Celery)
    ↓
AI Engine (Gemini/GPT)
    ↓
RAG System (Vector DB)
    ↓
Result Processing
    ↓
Supabase DB 저장
    ↓
CRM (결과 반환)
```

- **데이터 입력**: 파일 업로드, 텍스트 입력
- **HQ API 호출**: RESTful API 호출
- **결과 표시**: PDF 다운로드, 그래프 렌더링

#### 주요 컴포넌트
1. **카테고리 선택 박스** (상단)
2. **좌측 Input 패널** (파일 업로드 + 텍스트 입력)
3. **우측 Output 패널** (AI 결과 표시)
4. **로딩 인디케이터** (백그라운드 연산 중)

## 7. 기술 스택 및 아키텍처 요구사항

### 7.1 HQ 앱 (Backend)
- **언어**: Python 3.11+
- **프레임워크**: FastAPI (RESTful API)
- **AI 엔진**: Gemini API, GPT-4 API
- **RAG**: Pinecone/Weaviate + LangChain
- **작업 큐**: Celery + Redis
- **데이터베이스**: Supabase (PostgreSQL)
- **스토리지**: GCS (Google Cloud Storage)
- **배포**: Cloud Run (GCP)

### 7.2 CRM 앱 (Frontend)
- **언어**: Python 3.11+
- **프레임워크**: Streamlit
- **HTTP 클라이언트**: `requests` 또는 `httpx`
- **파일 업로드**: `st.file_uploader`
- **PDF 생성**: `reportlab` 또는 `weasyprint`
- **배포**: Cloud Run (GCP)

### 7.3 데이터 흐름 보안
- **암호화**: Fernet (AES-128-CBC)
- **인증**: HMAC-SHA256 토큰
- **SSO**: CRM ↔ HQ 토큰 기반 인증
- **RLS**: Supabase Row Level Security
